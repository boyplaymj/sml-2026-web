## 0. 一句話定位

**神社 = 運氣值中樞。** 玩家花牙齒參拜/抽籤/請御守 → 得到「時限祝福 buff」→ 其他遊戲讀取運氣值當作機率/收益修正 → buff 會過期 → 自然回訪循環。

## 1. 核心心法：時限祝福制

運氣**不是**永久累積的死值，而是**會過期的祝福 buff**（仿真實御守一年效期、參拜一時加護）。這保證回訪。

### 1.1 運氣的六個子屬性（正典分類）

籤詩分項、御守、遊戲關聯全部收斂到這六軸。每軸 baseline = 50，範圍 0–100（含 buff 疊加可暫時超過）。

| 子屬性 | 對應籤詩分項 | 影響遊戲（需隨機性者） |
|---|---|---|
| 💰 **財運** | 商賣・金運・相場 | 麻將館收益/客流、股市操盤、火車大亨貨運結算、競猜分紅 |
| ⚔️ **勝運** | 爭事・勝負 | 賓果、PK 罰球、UPW、骰寶/BJ 等 PvE 機率（**PvP 只給彩蛋不動實際勝率**） |
| 📚 **智運** | 學問・仕事 | 每日任務/解謎獎勵權重、地震防災答題、成就進度 |
| 🩺 **body（厄除）** | 健康・病氣 | 消災解厄抵抗、捍衛路權護體機率、抵銷厄運負成長 |
| 💗 **人緣** | 戀愛・縁談・待人 | 社交任務、投票應援、生日互動 |
| 🧳 **行運** | 旅行・失物・轉居 | 捍衛路權、火車大亨在途事件、開箱/隨機事件好結果機率 |

> 另有一個 **綜合運** = 六軸加權平均，用於「沒有明確對應」的通用隨機（開箱、隨機事件整體好壞）。

### 1.2 運氣如何影響遊戲（數值化，保守）

各遊戲讀取相關子屬性，套用**溫和**修正，避免課金（花牙齒）碾壓：

- **收益型**（財運）：`收益倍率 = 1 + (財運-50)/500` → 極值僅 ±10%。
- **機率型**（勝運/綜合運，僅 PvE）：好結果權重 `w' = w × (1 + (值-50)/250)` → 極值 ±20%。
- **抵抗型**（body）：厄運負成長速率 `× (1 - (body-50)/200)`。
- **🛡️ 鐵律：PvP 對戰（PK/UPW/賓果對抗）只吃「彩蛋/風味」修正，實際勝負判定不受運氣影響。** 防止 pay-to-win。
- 每個遊戲的實際係數 Phase 2 wiring 時逐一調校並記在案；Phase 1 先把運氣值算出來、可查詢。

## 2. 抽籤系統（おみくじ，仿真實神社）

### 2.1 籤階（11 個常規階 + 大大吉彩蛋）

11 常規階：`大吉 → 吉 → 中吉 → 小吉 → 末吉 → 末小吉 → 凶 → 小凶 → 半凶 → 末凶 → 大凶`（`config.omikujiWeights` / `rankTtlMultiplier` 皆對齊這 11 階）。另有超稀有 **大大吉** 彩蛋（預設不在權重內、抽不到，留給活動觸發）。

各階**權重可後台調**（煞車）。預設大吉稀有、吉/中吉/小吉為大宗、大凶極稀有。

### 2.2 一張籤的結構

```jsonc
{
  "id": "omikuji-0007",
  "rank": "中吉",
  "waka": "…一句和歌／詩…",         // 預寫，仿真實籤
  "sougou": "…總合運勢文字…",
  "items": {                        // ← 分項，各對應一個子屬性
    "商賣": { "text": "…", "score": +8 },   // → 財運 +8
    "健康": { "text": "…", "score": -3 },   // → body -3
    "戀愛": { "text": "…", "score": +5 },   // → 人緣 +5
    "學問": { "text": "…", "score": +2 },   // → 智運 +2
    "旅行": { "text": "…", "score": 0  },   // → 行運
    "爭事": { "text": "…", "score": +4 }    // → 勝運
  }
}
```

- **抽到籤 → 依 items.score 給對應子屬性一段「祝福 buff」**（含時效，見 §1）。籤階越高、各項 score 越正。
- **凶籤**：可到境內「結ぶ」把凶籤綁起來化解（真實習俗），或不綁則帶著負 buff。
- **籤詩來源＝預寫籤池（$0）**，仿真實神社文本手工撰寫數十張，不走 LLM 即時生成（省成本）。存 DDB，可後台增修。

## 3. 御守（お守り）— 攜帶型 buff 修正器

比照籤詩分軸。每種御守長期提升某個子屬性。

| 御守 | 提升 | 遊戲效果例 |
|---|---|---|
| 金運守 | 財運 | 麻將館/股市收益 +X% |
| 勝守 | 勝運 | 賓果/PK/UPW 玄學（PvE 機率、PvP 彩蛋） |
| 學業守 | 智運 | 任務/解謎獎勵權重 |
| 健康守 | body | 厄運抵抗、路權護體 |
| 縁結守 | 人緣 | 社交/投票/生日 |
| 交通安全守 | 行運 | 捍衛路權護體、火車在途 |

### 3.1 效期與回收（核心留存機制）

- **御守效期 = 365 天**（仿真實一年）。
- **到期未回收到「古札納所」→ 御守變「穢れ」→ 開始增加厄運（幸運值負成長）**：每日扣 body/綜合運。持有越多過期御守，扣越兇。
- **玩家如何得知？** 透過**神職人員的對話文字**提示（不是硬跳系統警告）：接近到期時境內 NPC 會旁白提醒「御守已滿一年，該還納了…」；過期後語氣轉為「身上纏著舊守的穢れ呢…」。
- **回收 → 消除負成長 + 給「功德值」**（回收獎勵，可折抵新御守/御朱印）。
- 這條 = 一年一循環的長線回訪鉤。

## 4. 消災解厄（御祈禱）— 直接綁生日功能

### 4.1 資料已就緒

生日系統存**完整 `yyyymmdd`**（例 `19940807`，含出生年，程式已能算歲數）。因此**真實厄年表（数え年 = 満歲+1）完全可行**。

### 4.2 厄年表（数え年）

- **男 本厄**：25、42（大厄）、61
- **女 本厄**：19、33（大厄）、37、61
- **前厄** = 本厄−1，**後厄** = 本厄+1（各一年）

### 4.3 機制

- 玩家處於厄年（前/本/後厄）→ 該年套一個**厄 penalty**（body/綜合運長期負修正）。
- 到「御祈禱受付所」花牙齒請神職人員**消災解厄** → 化解／減輕該 penalty，發護符。
- **與生日快樂功能聯動**：既有生日 timer 觸發時，一併算 `数え年`，若逢厄年 → 在生日祝賀中附「今年是你的○厄，要不要到神社除厄？」的鉤子。
- 未填生日者：改用**生肖本命年**（只需出生年，或請補生日）作 fallback。

### 5.2 奧社「聽牌試煉」——翻牌湊聽牌小遊戲（v0.2 定案：memory-flip 版）

奧社入口不是花牙齒就能進——要通過牌神的**聽牌試煉**：一個「翻牌記憶 × 湊聽牌」的地城式回合遊戲。**引擎抽成可重用 module，日後可脫離神社獨立開放為小遊戲。**

**核心循環（每關）**
1. 起程前置：當日已手水；在開放時間內（§5.3）；付一次「入山料」牙齒。
2. **宣告聽牌目標**（例：聽三六餅）。
3. **盒子 = 蓋著的牌方陣（🀫）**：內含「一副能聽中目標的正解牌」＋若干**干擾牌**，洗亂後全部蓋住。
4. **回合**：翻開一格 →
   - **取** → 收進手牌（該格清空）；
   - **放回** → 蓋回去（你已看過＝記憶點）。
5. 手牌湊到該關需求張數 → **上牌**：聽牌 solver 判定「是否**正好**聽目標」。
   - **對** → 過關，得 🦷（第 8/9/10 關再加 ⭐經驗）。
   - **錯/多** → 可**捨牌**把一張放回盒子（耗一回合）再翻再湊。
6. 連過 **10 關** → 抵達奧社內殿：可行**深度參拜**（祝福更強、稀有籤率高）＋購買 **奧社限定御守／御朱印**（只有這裡買得到＝稀有收藏 sink）。

> **34 種牌不必列**：盒子全是相同的牌背 🀫，一次只翻一張；畫面同時只出現「翻開的那張 ＋ 你的手牌」。

**難度模型（四軸，L1→L10 遞增，全 config per-level 可調）**

| 軸 | 易 → 難 | 機制 |
|---|---|---|
| **A 聽牌集合大小** | 2 張（兩面 三六餅）→ 3 張（三面 三四五）→ 4 張＋（多面 三四五六） | 目標越寬，需越複雜牌型才「正好聽這些」；牌型複雜度（單騎/邊張/嵌張/兩面/多面/複合）與此連動 |
| **B 盒子花色數** | 三花（萬筒條）→ 兩花 → 單花（只萬子） | 同花無法用花色排除、干擾牌更相近，記憶與辨識更硬 |
| **C 解的數量** | 多組解（4446/4455/4666 皆可、可改道）→ **唯一解**（盒子只放其中一種組合） | 沒有備援路線，非找到那組不可 |
| **D 記憶/運氣壓力** | 小盒、少干擾 → 大盒、干擾多且接近、**後段加翻牌次數上限** | 地城式翻找的運氣＋策略張力 |

**命門技術（純邏輯、可單測、$0 無 LLM）**
1. **聽牌 solver＝重用既有麻將核心，不重造**：台灣麻將胡/聽判定地城工程師已寫好（`Common/ReadyHandLogic.js` 的 `checkHuPai`/`isCanHU`），抽成共用純模組 `Common/MahjongHand.js`（規格見 **`tools/mahjong-hand-core/SPEC.md`**）。驗答用 `waitingTiles(手牌)`＝集合**正好等於**目標（exact）。手牌張數契約：3k+1 判聽、3k+2 判胡；台灣 16 張制免役即胡。
2. **出題器＝重用既有花色出題**：`ReadyHandLogic.createBaseCards(color)` 已支援 `m/s/p/ms/mp/sp/msp` 花色約束（正對難度軸 B）；`ReadyHand` 遊戲已預生成題庫存 `sweetbot-readyhand-topic`（`level=聽幾張`）→ 奧社可沿用題庫或出題器。⚠️ **軸 C（唯一解）需額外驗證**：確認盒子內**湊不出第二組**能聽中目標的牌（盒子小，可窮舉）。→ 設計冊只寫難度軸參數，**不寫死每關手牌**（solver/出題器為唯一真理）。
3. **對局狀態持久**：盒子佈局要在按鈕之間保留 → 用**帶 TTL 的對局表**（仿牙菌斑 battle 租約），結束/逾時自動清，不污染永久資料。
4. **呈現**：🀫 方陣用 Discord 按鈕（上限 5×5＝25 格，正好夠大盒子）；翻開的牌＋手牌用既有麻將 emoji 管線。

**其他**
- **狀態欄位**：玩家奧社進度／當日是否已通關／通關效期存 `fortune` 表（`okumiyaClearedAt`/`okumiyaAttemptDate`/`okumiyaAttemptsToday`），lazy 判定；當前對局盤面存 TTL 對局表。

- 通關另給**功德值／成就**，強化榮譽感。

- **經濟量級**：全清 ~1170🦷 ≈ 人均日收 1/3（見 [牙齒經濟基線]）；有時間閘＋入山料＋每日上限＋純技術門檻四重煞車，且後台可整體調降。屬「硬核榮譽關」而非日常印鈔。
- **軸 C（唯一解）落地**：出題器生成後，用共用核心窮舉檢查「盒內湊不出第二組聽中目標的手牌」；多解/少解則放寬。
- **翻牌上限**：L1–4 無限（純記憶入門）；L5 起設限且 slack 逐關收窄（如 L10 收集 13 張、上限 20 → 僅 7 次容錯）。

> **仍待你拍板**：整體數值鬆緊（🦷/⭐ 要不要加碼或降）、入山料價、每日上限、捨牌規則（目前 L9 限 1 次、L10 不可）。以上為**提案預設**，後台皆可調。

### 5.3 開放時間閘（v0.2 新增）

仿真實神社「參拜時間 / 受付時間」。時間一律以**台北時區（UTC+8）** 判定，**時段全部寫在 config、後台可調**。

| 行為 | 開放時間（預設） | 關閉時的表現 |
|---|---|---|
| **參拜系**（主殿參拜、奧社試煉/參拜、御祈禱除厄） | **09:00–17:00** | 神職人員旁白「本日參拜已結束，明朝請早」；按鈕 gate 擋下 |
| **御朱印**（受付/蓋印） | **09:00–15:00**（15 時受付終了） | 「御朱印受付已於 15 時結束了呢」 |
| 授與所（買御守/神札）、古札納所（回收）、繪馬、手水舍、石柱 | 全日開放（可 config 各自設時段） | — |

- 判定：`config.hours`（如 `{ sanpai:[9,17], goshuin:[9,15] }`，24 小時制、台北時區）。純比較現在的台北小時，無背景 job。
- 邊界：以「起始動作時刻」判定；已進行中的流程（如試煉參道走到一半）不因跨過 17:00 而中途踢出（避免體驗斷裂）＝**進場檢查、不中途踢**。
- fail-safe：config 缺 `hours` → 視為全日開放（不因設定缺失把玩家鎖在外）。

## 6. 資料模型（DDB，全 PAY_PER_REQUEST）

> 表名前綴一律 `sweetbot-shrine-*`（對齊 sweetbot-next 既有慣例，非 `sml-`）。全 `PAY_PER_REQUEST` @ `ap-southeast-1`。詳細 schema/keys/存取模式見 `STAGE0.md`。

| `sweetbot-shrine-ema` | 繪馬牆 | PK=月分桶, SK=createTime；玩家願文（公開）、讚 |
| `sweetbot-shrine-pillar` | 表參道石柱捐獻 | PK=discordId（累計）；金額、刻字（榮譽榜=小表 scan） |
| `sweetbot-shrine-omikuji-pool` | 籤詩池（後台可增修） | PK=omikujiId；籤階、和歌、分項 score（見 §2.2） |

**核心零 GSI**（對齊牙菌斑原則）：所有查詢都是 by-player（GetItem/Query PK）或小表 scan，Phase 1 不需任何 GSI。若 Phase 4 後台要「即將到期御守全服廣播」或「石柱金額排行」再按需加。奧社對局表是唯一開 **TTL** 者（暫態盤面），對齊牙菌斑「battle/world 才開 TTL、核心永久表不開」鐵律。

> 運氣「即時值」採 **lazy compute**（讀取時依 buff 陣列的 `expireAt` 現算，不跑背景 job；對齊牙菌斑/報稅的隨賺隨記與惰性結算原則）。過期御守的穢れ負成長也在讀取時結算。
