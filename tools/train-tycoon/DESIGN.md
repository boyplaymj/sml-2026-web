# 甜甜火車大亨 — 設計冊 v0.1

> 類型:即時經營／物流調度模擬(策略模擬,致敬《鐵路大亨 Railroad Tycoon》/《A列車》/《Transport Tycoon》)
> 平台:甜甜 Discord bot(MVP);玩法驗證後再評估搬兩雀 APP
> 日期:2026-07-15 起草(Claude 設計,待使用者拍板 → Codex 驗證/實作)
> 姊妹作:[[project_mahjong_tycoon]](同為牙齒經濟經營模擬,可重用 tick/惰性結算/後台型錄架構;本冊沿用其正典結構)

---

## 0. 定位一句話

每位玩家擁有**自己的火車站**,靠**購車編組 → 開貨運/客運路線 → 排班次調度 → 在途行駛(需時間)→ 抵達結算**,把牙齒滾大,擴站升級、經營越大越強。

**核心差異(vs 麻將館大亨):** 麻將館是「固定店面、路過客上門」;火車大亨是「主動派車、跨節點運送、行駛要時間」——**物流路網 + 班次調度**是心臟,天生的「掛機式」idle 收益(你排好班次,離線時列車照跑照賺)。

---

## 1. 核心迴圈

```
建站 → 開路線(貨運線 / 客運線) → 購車 + 編組(車廂節數) → 排班次 / 派送
   → 列車在途(需行駛時間) → 抵達結算(貨物價差 / 票價收入)
   → 再投資(擴站 / 升級 / 設備 / 請人) → 應付事件(誤點 / 故障 / 需求波動) → 擴大路網
```

存活/經營越久 → 解鎖更大機車頭、更多月台、更遠更肥的目的地 → 正回饋。

**兩具收益引擎(玩家可只玩一種或雙修):**
| 引擎 | 玩法 | 主要決策 |
|---|---|---|
| **�freight 貨運** | 目的地有貨物需求/報價,裝貨→派車→行駛→交貨賺價差 | 載什麼貨、幾節車廂、何時發、走哪條線、接不接急件(時限) |
| **🚉 客運** | 開客運班次(路線+時刻表+票價),乘客上車付車資 | 票價高低(需求彈性)、班次密度、車廂座位數、覆蓋哪些尖峰 |

## 2. 世界與路網

### 2.0 世界模型(定案:共用互連路網)
- 路網 = **全服共用**,節點分兩類:
  - **玩家車站**:每位玩家一個(可被別人開路線連過去/送貨/載客)。
  - **NPC 城**:系統節點,固定/波動的貨物需求與客運需求,永遠有得跑(保證少人時也有內容)。
- 路線 = 節點對之間的邊;你可開「你的站 ↔ NPC 城」或「你的站 ↔ 別人的站」。
- **分階段點亮**:Phase 0~2 先只跑「玩家站 ↔ NPC 城」把核心迴圈站穩;**Phase 3 點亮玩家↔玩家路線**(互送貨/互載客、同線競爭分食)。world 表結構從一開始就按共用設計,不用改結構。

### 2.1 目的地節點(destinations)
- MVP 先 3~4 個 NPC 城 + 各玩家自己的站;之後可擴城、Phase 3 開放玩家站互連。
- 每個目的地屬性:
  - `distance`:距離 → 決定**行駛時間**與**燃料成本**(遠=賺得多但慢、耗油、佔用月台久)
  - `freightDemand{}`:各貨物的需求量與**報價**(隨世界 tick 波動;供過於求會跌價)
  - `passengerDemand`:客運需求池(受你票價/班次密度分食)
  - `unlockTier`:要車站升到某等級才能通(遠城/大城後期解鎖)
  - `heat`(選配):熱度,節慶/賽事會拉高需求

### 2.2 時間模型 = 在途 + 惰性結算(idle 核心)
- **列車在途**:一次派送 = `departAt` + `travelDuration(distance, 機車頭速度, 車廂載重)` → `arriveAt`。抵達才結算收入。
- **惰性結算(lazy tick)**:玩家一互動,用 `now - lastSettledAt` 一次補算這段期間**所有已抵達的列車**的收益/成本/事件。省算力、天然掛機。
- **世界 tick(全服低頻,5–15 分)**:只做——刷新目的地需求/報價波動、擲隨機事件、推進節慶熱度、(有 @通知的)抵達提醒。
- **客運班次自動重發**:排好時刻表(每 X 分一班)後,離線期間列車**自動循環發車**,惰性結算補算票收——這是主要 idle 收益來源。

### 2.3 時段尖峰(借麻將館大亨 §2.4)
- 客運需求隨遊戲內時段起伏(通勤尖峰↑、深夜↓);貨運需求也可有旺季/節慶。
- 惰性結算按經過的時段積分,不用平均值硬算。
- 玩家用**班次密度 + 發車時刻**去對準尖峰;尖峰沒班次 = 把乘客讓掉。

## 3. 車站經營維度(玩家能調的旋鈕)

對應你列的需求逐項落位:

| 你說的 | 系統維度 | 影響 |
|---|---|---|
| **火車班次** | 客運線時刻表(每 X 分一班 / 發車時段) | 覆蓋多少乘客尖峰 vs 燃料+磨耗+月台佔用 |
| **送貨時間** | 貨運派送時機 / 接急件時限 | 準時交貨=獎金;逾時=賠償;旺季報價高時發車=賺更多 |
| **車廂節數** | 每列編組(貨:載重噸位 / 客:座位數) | 單趟運能↑,但車重→行駛變慢、耗油↑、要更強機車頭 |
| **車站人員** | 調度員/站務/維修/售票/司機 | 同時可跑的路線數、準點率、故障率、乘客攬收、吞吐 |
| **票價** | 客運票價 | 高票價賺多但需求↓(彈性);低價衝量 |
| **車站大小擴充** | 月台數 / 貨場容量 / 候車廳 | 同時在途列車數上限、貨物暫存量、乘客吞吐上限 |
| **設備添購** | 號誌/機廠/裝卸吊車/加油站/售票機… | 準點、故障率、裝卸速度、燃效、乘客攬收(見 §3.2) |
| **車站升級** | 車站等級 tier | 解鎖更大機車頭/更多月台/更遠目的地/更肥合約 |

### 3.1 車輛與編組(fleet)
- **機車頭(locomotive)**:等級決定**牽引力(可拉幾節)、速度、燃效**;一次採購 + 每趟燃料 + 折舊/故障。
- **車廂(cars)**:貨車廂(載重)/客車廂(座位),按需編組;節數越多運能越大但越重越慢。
- 一列車 = 1 機車頭 + N 車廂,受**車站月台/機車頭牽引力**上限約束。

### 3.2 設備採購(一次性資產 + 維護費)
- `號誌系統`→ 提高同時在途列車上限 + 準點率
- `機廠/維修庫`→ 降故障率、故障修復快
- `裝卸吊車`→ 貨物裝卸更快(縮短站停時間)
- `加油/加煤站`→ 燃效↑(每趟省油)
- `售票機/候車廳升級`→ 乘客攬收率↑
- 部分有每 tick 維護費/水電;有折舊/故障隨機事件掛鉤。

### 3.3 人員 / 排班
- 職務:**調度員**(可同時管理的路線數↑)、**站務**(吞吐)、**維修工**(故障率↓)、**售票員**(客運攬收↑)、(可選)**司機**(每列車需司機才能發,簡化版可省)。
- 各有等級/薪資/士氣;薪資持續計費(牙齒 sink)。

## 4. 收益與支出(結算)

```
收入 = 客運票收 + 貨運交貨價差 + 急件獎金 + 長約分紅 + 節慶爆量
支出 = 燃料 + 薪資 + 設備維護 + 車站維護 + 逾時賠償/退票 + 升級/採購(一次性)
```
- 現金流長期為負且牙齒見底 → **危機**;可低成本重整(保留等級,精簡車隊)。避免像麻將館「倒店」那麼硬,火車以「縮編求生」較貼題。
- **防通膨**:燃料/薪資/維護/升級是天生大 sink,收支自平衡(同麻將館思路)。

## 5. 隨機事件(二選一分支,借麻將館 §7 架構)

觸發條件 + 情境 + 2 選項 + 結果。範例池:
- **誤點**(尖峰/惡劣天氣更常見)→ 賠票 or 加班趕點
- **機械故障**(維修等級低更常見)→ 停駛維修 or 帶病硬跑(風險)
- **暴雨/雪封線**→ 停開該線 or 改道(繞遠)
- **旺季爆量需求**(節慶)→ 加開班次搶收 or 維持不超載
- **VIP 包車**→ 接(大收入,佔用車)or 拒(保住常客)
- **貨物損壞/遺失**→ 理賠 or 追查
- **油價波動**→ 影響全隊燃料成本
- **名人搭乘(SML 選手彩蛋)**→ 聲量/需求波動大

## 6. 難度曲線與求生

- 燃料成本、薪資隨規模上升;遠程目的地報價高但風險/耗時大 → 逼玩家平衡短線穩收 vs 長線豪賭。
- 目的地需求會**波動甚至崩盤**(某貨物大家都在送 → 報價跌)→ 逼玩家分散路線、追旺季。
- (若走互連模式)玩家多了同線競爭分食客貨,天然對抗張力。

## 9. 資料結構(DynamoDB 草案,全 PAY_PER_REQUEST)

- `train-tycoon-config`(PK=section)：後台設定源頭(draft/published)。sections=`catalogs`(locomotives/cars/equipment/staff 型錄)、`destinations`(目的地城市)、`balance`(需求彈性/燃料/事件機率/時段係數)。
- `train-tycoon-stations`(PK=userId)：車站狀態、金庫參照、tier、platforms、yard、waitingHall、`equipment[]`、`staff[]`、`fleet[]`(機車頭+車廂)、`lines[]`(客運/貨運路線+時刻表)、`lastSettledAt`。
- `train-tycoon-transit`(PK=userId, SK=arriveAt#dispatchId)：在途/待抵達列車(裝載內容、目的地、發車/抵達時刻)。
- `train-tycoon-world`(PK=nodeId)：全服共用目的地世界狀態(需求/報價/熱度)。若走純孤立模式(決策②A)此表可省。
- `train-tycoon-events`(PK=userId, SK=ts)：事件流水/待回應事件。

## 11. MVP 切期(每塊 <25 分工作量,分階段驗;見 [[feedback_work_chunking]])

- **Phase 0 — 核心手感**:資料表 + 單一指令面板 + 建站 + 買 1 機車頭+貨車廂 + 1 目的地 + 手動派送 → 惰性抵達結算賺牙齒 + 提款。(先驗「派車→等→收錢」手感,**不含**客運/升級/事件)
- **Phase 1 — 客運 idle**:客運線 + 時刻表自動發車 + 票價需求彈性 + 離線班次結算(掛機收益的鉤子)。
- **Phase 2 — 經營深度**:車站擴充(月台/貨場/候車廳)+ 升級 tier + 設備採購 + 人員/調度 + 準點/故障/燃料。
- **Phase 3 — 世界路網**:多目的地共用需求波動 + 貨運急件合約 + 報價崩盤張力。
- **Phase 4 — 事件系統**:隨機事件表(誤點/故障/天氣/旺季/VIP/名人彩蛋)。
- **Phase 5 — 平衡 + 經濟串接**:數值調校 + 牙齒經濟後台串接 +(選配)玩家站互連/對抗。

> 註:客運(Phase 1)刻意早於經營深度——客運時刻表是最強的「掛機」鉤子,先讓玩家嚐到離線也在賺,再堆深度。

## 12. 決策(已拍板 2026-07-15)

1. **名稱/指令**:入口指令 `!火車`(定案);遊戲顯示名沿用「甜甜火車大亨」。
2. **世界模型**:**共用互連路網**(見 §2.0)——玩家車站 + NPC 城,路線可連玩家↔NPC、也可玩家↔玩家。Phase 0~2 先跑玩家↔NPC,**Phase 3 點亮玩家↔玩家**;world 表從頭按共用設計。
3. **Phase 0 引擎**:先**貨運**(派車→交貨因果最直觀);客運排 Phase 1。
4. **視覺**:**embed + emoji 拼貼一張情境圖 + 互動按鈕**(定案,見 §8),沿用甜甜 emoji 拼圖產線。
5. **直播聯動**:**先把遊戲做出來,賽事熱度聯動後補**(列後期彩蛋,不進 MVP)。
6. **車種**:機車頭 tier + 車廂類型草案見 §13,**討論中**,敲定後入型錄。

| **コキ** | 貨櫃車(container) | 大宗貨櫃,單節載量最大 |

**客車廂**:普通車 / **グリーン車**(綠色車廂=商務,客單價高) / **観光列車**(景觀車廂,如リゾートしらかみ風,觀光/景點線需求加成)。

> tier 數(先給 6 級)、牽引/速度/價格曲線為草案,實際數值待 Phase 0 平衡試玩後填入 `catalogs`;新幹線動力車組(N700S/E5/L0)本質是「編成」,遊戲內簡化為「動力車 + 客車廂」抽象。真實 JR 型號僅作社群同好風味使用。

## 14. 車站視覺呈現系統(使用者最在意的核心體驗)

> 面板打開第一眼 = **自己的車站側視拼貼圖**;花牙齒的每個升級都要「看得見」——越大越豪華,堆收藏/炫耀感。這是留存主軸,也反向決定 `stations` 要存哪些欄位。

### 14.1 呈現形式
- embed 主圖 = 一張**側視「站前風景」PNG**,由**格點畫布分層合成**(沿用甜甜 emoji 拼圖產線 [[reference_sweetbot_emoji_pipeline]] / 動態版 [[reference_animated_gif_pipeline]];參考捍衛路權的場景合成)。
- 畫布**寬度隨 tier 增長**(Tier1 窄 → Tier6 寬);內容由車站狀態即時組出,不是固定圖。
- MVP 可先用「emoji unicode 排版 or 少量素材」快速出圖,Phase 2 再換金庫高清素材/加動態(列車進站動畫)。

### 14.2 分層畫布(每層由狀態驅動)
| 圖層 | 由什麼欄位決定 | 隨升級變化 |
|---|---|---|
| ① 天空 | 遊戲時段 / 天氣事件 | ☀️白天 / 🌙夜 / ⛈️暴雨 / ❄️雪 / 🎆節慶 |
| ② 天際線建築 | `tier` + `decor[]` | 🏚️小屋→🏤站房→🏢🏬大樓→🏙️🗼天際線 |
| ③ 站體正面 | `tier`(換 sprite) | 木造→磚造→玻璃帷幕→巨型航廈 |
| ④ 月台+軌道 | `platforms`(擴充)+ `fleet[]`(停靠車) | 1→5+ 月台;停放你買的 JR 車種 emoji |
| ⑤ 前景裝飾 | `equipment[]` / `decor[]` | 🕐時鐘 / 🚦號誌 / 🏗️貨場吊車 / ⛲噴水 / 🌷花圃 / 🚕計程車招呼站 / 👥乘客 |

### 14.3 tier 站容外觀階梯(草案)
| tier | 站名風格 | 月台上限 | 站房 sprite | 解鎖裝飾 |
|---|---|---|---|---|
| 1 | 無人小站 | 1 | 🏚️木造 | (無) |

| 4 | 主要駅 | 4 | 🏢 | 🌃夜景燈 / 🏗️貨場 |

| 6 | メガターミナル | 6+ | 🗼🌆 | 全解鎖·專屬豪華飾件 |

### 14.4 driven-by 欄位彙整(→ §9 `stations` schema)
車站 item 要能組出畫面,至少存:`tier`、`platforms`、`fleet[]`(各車 type+數量,決定月台停哪些車)、`equipment[]`、`decor[]`、(可選)`theme`(季節/節慶皮膚)。渲染器 = 純函式 `renderStation(stationState) → PNG`,無副作用,好測。

### 14.5 開發切點

- **Phase 2**:接金庫高清素材 + 裝飾圖層 + tier 站房 sprite 全套。
- **Phase 3+**:列車進站/發車動態 GIF(用動態 GIF 管線)。

### 14.6 呈現層定案(2026-07-15,重要)
- **硬性原則:一切在 Discord Embed 內完成,不跳外部網頁、不做 Activity。** 玩家永遠留在 Discord。
- **呈現 = emoji/tile 拼貼 + 即時疊圖生成一張圖(PNG,可選動態 GIF)+ 按鈕操作**;依 §14.4 狀態簽章快取、sharp 合成。
- **排版**:可採 **Components V2**(2025 原生 embed 元件:media gallery/container/section,仍全 in-embed)把「大圖 + 數據卡 + 按鈕分區」排得更精緻;需確認甜甜 discord.js ≥14.16。
- **畫風定案(2026-07-15,最終):B 斜角 isometric + 16-bit 像素風 + 動圖(火車會動)。**(曾一度改 32-bit RRT 細膩、AI 實測很成功,但為**生成時更易統一風格**改回 16-bit 像素。)
- **一致性關鍵手法(像素風的真正優勢)**:AI 原生像素圖常「不夠像素」(格亂、反鋸齒),但可用**後處理正規化**機械性統一——每張都過同一道「**去背 → 降到固定低解析度 → 套同一組共用調色盤(shared palette, ~24 色)→ nearest 放大**」→ 不管 AI 生什麼,最後都被壓成同一像素網格、同一組顏色。工具:`tools/train-tycoon/mockup/pixelize.py`(共用調色盤由全體 sprite 不透明像素 median-cut 得出,再 remap 每張)。POC:`/tmp/train-px/panel_px.png`。
- **在 embed 內無法做的**:拖曳/縮放/點圖等「圖上互動」→ 一律用**按鈕/選單**替代(升級/派車/擴站)。斜角只是「畫風」,不等於「可互動地圖」。
- **架構硬性含意**:車站會隨 tier/月台/車隊/裝飾**重組**,所以**不能用「AI 生一整張圖」**(無法組合)→ 必須是**模組化像素 sprite 庫**(等距地面 tile/軌道各向/建築各 tier/列車含動畫幀/裝飾)由**程式合成 + 逐幀動畫 → GIF**,依 §14.4 狀態簽章快取。**「這套 16-bit 等距 sprite 庫」= 本專案最大的一次性美術工。**
- **sprite 取得路線(定案:AI 生成)**:免費 CC0 等距鐵道包實測=不是像素風就是低模 3D,**無 32-bit RRT 細膩 2D**;→ 改走 **Bedrock SD3.5 生 hero sprite(火車/站房)**,實測**完美打中 RRT 細膩斜角、火車天生斜角**(POC:`/tmp/train-ai/panel_ai.png`)。管線=SD3.5 生 → PIL flood-fill 去背(邊緣連通、保白煙,tools/train-tycoon/mockup 的 bg_flood.py)→ 合成(compose_ai.js)。**去背 remove-background Bedrock 模型需 inference profile 不能 on-demand,故用 PIL 去背。**
- **AI 素材路線的真工(新美術工作型態)**:從「畫」變「**下 prompt + 篩選 + 顧一致性**」——① 一致性(每次生成風格/光照/比例/主題會飄,站房曾生成偏東方塔樓味,要用固定 prompt 模板/風格參考/多抽篩)② 地面/軌道 tile 的無縫拼接 AI 較弱 → hero 物件走 AI、地面可程式/tile 包 ③ 火車多朝向要分別生(同車一致性靠 prompt)④ 動畫=sprite 沿軌位移 + 煙霧疊層(簡單);逐幀變形較難。
- **成本**:sprite 是**一次性烘焙**(生成 + 去背後存檔),**遊戲執行期不呼叫 LLM/生圖**;生成成本=Bedrock 每張幾美分、總計幾美元一次性(見成本控管段)。
- **動畫來源**:列車位移逐幀、車輪/冒煙/號誌閃爍;用既有動態 GIF 管線 [[reference_animated_gif_pipeline]](/tmp gifwrap、發前解回驗證)。
- **效能**:GIF 比 PNG 大且生成慢 → 狀態簽章快取更關鍵;主面板才用 GIF,子頁用靜態或純文字。

#### 已評估並排除的呈現方案(勿重議)

|---|---|
| Discord **Activity**(Embedded App SDK) | 工程大一級距;**未認證只能 <25 人群/DM**,上大群要 App 驗證(Stripe 實名)+ 隱私政策/ToS 認證。詳見 [[ACTIVITY_EVAL.md]] tools/train-tycoon/ACTIVITY_EVAL.md。使用者要「全在 embed」故不採。 |
| **伴生網頁**(一般連結按鈕→瀏覽器互動斜角地圖) | 能拿 Activity 90% 視覺又免認證,但**會跳出 Discord 到瀏覽器分頁**,不符「全在 embed」硬性原則。故不採(但技術上是 Activity 的墊腳石,保留為未來選項)。 |

## 15. 玩法定案 v0.2（2026-07-15 共同設計，取代前文緊繃/風險框架）

> 本節是與使用者逐輪拍板的**現行定案**;凡與 §2.0 / §5 舊敘述衝突,以本節為準(舊文保留作沿革)。定調:**療癒系送貨 idle,概念要非常簡單、不緊繃**。

### 15.1 核心迴圈（定案）
> 從**車庫**挑車 → 拉到**月台**編組 → 派車送貨,顯示預估抵達 → 車在途數十分~數小時,期間隨機獎勵按頻率零星跳 → 抵達,基本運費/物資躺月台待領(永不消失) → 回來**領取**入金庫、騰月台 → 再派。

- **技巧 = 抓準送貨次數/時間**:車只有在途才賺、停站閒置不賺;抓準抵達回來即時再派+在場撿獎勵 = 最大獲利。鬆散玩仍有保底、零焦慮。**無任何懲罰,只有勤快多賺**。

### 15.2 兩種收穫（定案）
| | 基本運費/物資 | 隨機獎勵事件 |
|---|---|---|
| 來源 | 送達的任務本體收穫 | 途中偶發好事(順道載客/沿線打賞/名人搭乘/旺季爆量) |
| 時效 | **永不消失**,隨時領 | **限時 2 分鐘**,過了就沒 |
| 佔容量 | **是**,佔月台直到領取 | 否(接了入袋) |
| 在場需求 | 不用,惰性補算(離線安全) | **要**,需甜甜 @ping + 領取鈕(被 ping 到人在→一鍵領) |
| 觸發 | 每趟必有 | 長途途中**按頻率**零星觸發(非每趟固定;每趟/每小時設上限防洗版) |

### 15.3 兩面板兩畫法（定案）
- **月台**:斜角 iso 活躍場景(服役/停靠/發車的車 + 站體,美術主圖,見 §14)。
- **車庫**:**方格 grid** emoji 拼湊的車輛庫存(擁有但沒在跑的車頭/車廂,**不用斜角**,像收藏櫃)→ 省美術/算圖 + 給收集 JR 車爽感。
- **兩種容量、都靠擴站成長**:月台數 = 同時能跑幾趟(吞吐閘);車庫格 = 能擁有幾台車(收藏上限)。

### 15.4 世界模型改定：社交物流網（取代 §2.0 的分階段點亮）
- 玩家**建站時隨機 spawn 一個永久座標**;彼此距離固定且不同 → 距離決定**送貨時間 + 基本獲利**(近=快但低、遠=慢但高)。後期可花大錢**遷站**(牙齒 sink)。
- **基本獲利 = 距離 × 對方車站等級** → 送「遠程 + 高等級」玩家最肥。
- **互取貨閘**:對方沒取貨,你就不能送下一趟(給該收件人);月台被待領物資佔住 → 挑**活躍/會即時取貨/熟識朋友**當收件人 = 天然社交選擇壓力。
- **收件人取貨也拿獎勵** = 互惠飛輪(你送我我收我也賺,故有動機幫朋友取)。
- **NPC 城 = 冷啟動保底**:永遠在、**自動取貨**、但**獲利 capped 偏低**。取捨 = NPC 穩但賺少 / 真玩家賺多但要對方活躍。
- world 表存座標 + pairwise 距離。

### 15.5 防分身（定案：四層 + 疲勞軸，純經濟嚇阻不靠身分偵測）
四層疊加:①**結構天然**(獲利=距離×對方等級 → 分身 Tier1 本身低價值,要 farm 得先養高)②路線疲勞 ③短程疲勞 ④互取貨閘(每趟親手取 = 純勞力)。

**路線疲勞**(每對 `sender→recipient` 一個 `fatigue` 0–100,每送 +25、每小時 −10):
| fatigue | 獎勵 | 時間 | 整批沒收到 |
|---|---|---|---|
| 0–30 | ×1.0 | ×1.0 | 0% |
| 30–60 | ×0.7 | ×1.2 | 5% |
| 60–85 | ×0.45 | ×1.4 | 20% |
| 85–100 | ×0.25 | ×1.6 | 35% |

→ 連送 2 趟同一人 OK,第 3 趟起咬,隔 ~4 小時回滿。

**短程疲勞**(sender 全域):短程 = 距離後 25% 門檻;滾動 +1/趟、−1/30 分;同 window 前 3 趟全額,之後 ×0.6 → ×0.3;長途不受影響。

全部進 config `balance.antiAbuse{}`,後台可調 + kill switch;NPC 城也吃這兩套。

### 15.6 對資料結構的含意（待 Block A 之後同步）
- `stations`:加 `coord{x,y}`、`platforms`(月台格)、`depotSlots`(車庫格)、`fleet[]`(擁有車輛,在庫/在途狀態)。
- `transit`:每趟綁 senderId → recipientId、distance、預估 arriveAt、待領 payload、隨機獎勵 queue(各帶 2 分 TTL)。
- 新增 pair 疲勞:可存 `stations` 內 map `fatigue{recipientId: {v, updatedAt}}` 或獨立表;短程疲勞存 sender 端滾動計數。
- `balance.antiAbuse{}` + 收件人取貨獎勵係數入 config。

### 15.7 車站等級（主堡）成長線（定案 2026-07-15）
6 級,兼「利己(容量/車/範圍)」+「社交(當紅收件人)」雙動機。

| Tier | 站名 | 月台上限 | 車庫格 | 建地槽(設施總數) | 解鎖車頭 | 送貨範圍 | 收件人價值倍率 |
|---|---|---|---|---|---|---|---|
| 1 | 🏚️無人小站 | 1 | 3 | 4 | D51/キハ40 | 近程 | ×1.0 |
| 2 | 🏤町の駅 | 2 | 5 | 10 | +DD51 | +中程 | ×1.3 |
| 3 | 🏬地方都市駅 | 3 | 8 | 20 | +EF210/EH500 | +遠程 | ×1.6 |
| 4 | 🏢主要駅 | 5 | 12 | 40 | +N700S | +超遠 | ×2.0 |
| 5 | 🏙️大型ターミナル | 7 | 18 | 80 | +E5 | 全圖 | ×2.5 |
| 6 | 🗼メガターミナル | 10 | 30 | **150+（裝飾近乎無上限）** | +L0 磁浮 | 全圖 | ×3.0 |

- 「收件人價值倍率」= 獲利公式的 `對方等級` 係數 → 高 tier = 社交磁鐵(躺著收取貨獎勵 + 身分)。
- **升級 = 牙齒(主 sink)+ 活躍門檻(擋課金秒跳,護住經濟)**,兩者都達才給升:

### 15.8 設施 facilities（基地建設層，定案 2026-07-15）
**主堡(tier)= 上限閘 + 底盤**(決定能蓋哪些設施/每種最多幾座/地皮/站房主體外觀);**設施 = à la carte 花牙齒買、每座疊一個 sprite 上站景圖、各給小機能**。是 §14「每個升級都看得見」留存主軸的具體化。

| 設施 | 類型 | 機能(全正向,不碰減災) | 上圖 |
|---|---|---|---|
| 🚉 月台 | 機能·可疊 | +1 同時送車 slot(受 tier 上限) | 月台+軌道 |
| 📦 貨櫃區 | 機能·可疊 | +受貨區容量(收更多別人的貨) | 堆疊貨櫃場 |
| ⛽ 加油站 | 機能 | 每趟省油 → 淨利↑ | 加油塔 |
| 🏭 機廠 | 機能 | 縮短在途時間(車回更快) | 廠房 |
| 🚦 號誌樓 | 機能 | 隨機獎勵觸發率↑ | 號誌塔 |
| 🚻 廁所 | 人氣·裝飾 | +人氣 | 小屋 |
| ☕ 候車廳/賣店 | 人氣 | 被動小額牙齒/tick or +人氣 | 玻璃廳/商店 |
| 🌷🕐⛲ 花圃/時鐘/噴水 | 純裝飾 | 微幅人氣(炫耀為主) | 裝飾件 |

- **人氣 stat(定案走 B·輕量)**:所有人氣/裝飾設施加總 → 小幅提高**收件人價值**或**隨機獎勵機率**;刻意做小、當風味不當硬指標(蓋滿裝飾的漂亮大站 = 更受歡迎的收件人)。
- **月台/車庫改為「上限」**:tier 給上限,玩家一座一座蓋(每蓋都看得見),更有建設感。
- **固定位置版型(定案 2026-07-15)**:站區用固定 slot 版型,每種設施有預定槽位(第 N 條月台長在第 N 個位置);renderStation 照 `facility.slotId` 畫到預定座標,**無排版演算法**、整齊好維護。分區固定槽:月台排 / 貨場區 / 機能區 / 站前街景 / 天際線。
- **擴充天花板拉到「非常高」(定案)**:功能設施(月台/貨場/機能)有平衡上限(見 tier 表);但**裝飾/街景/天際線走近乎無上限**(建地槽 Tier6 起 150+,裝飾可持續堆)——最健康的長期牙齒 sink,老玩家永遠有東西可堆可炫耀,滿版都市拼貼是終局炫耀主軸。
- **效能取捨**:滿版(數十~上百 sprite + 寬畫布)主面板用**靜態 PNG**,動態 GIF 留給低 tier / 有車進站時刻;狀態簽章快取(站沒變不重生)是滿版可行的關鍵。
- **受貨區(inbound)獨立於月台(outbound)**:別人送你的貨進受貨區(貨櫃區擴容),塞滿只擋「別人送你」不擋「你自己出車」→ 當紅不卡自己。
- 成本含意:**每種設施一顆 iso sprite** → sprite 庫增長(靜態貼件,美術計畫內)。設施型錄入 config `catalogs.facilities`。

### 15.9 運費公式 + 編組層（定案 2026-07-15，貨種走 B 正向特化）
收件人只提供「距離 × 等級」兩個係數,其餘獲利由**你的編組**決定(不做收件人指定貨物)。編組 = 這端唯一策略深度。

```
運費 = Σ每節車廂( 載量 × 貨種價值係數 valueMult ) × 基準單價 baseUnitPrice
       × 距離係數  ( 距離 / distanceRefUnit )
       × 對方tier倍率 tierRecipientMult ( 再 × 人氣微調 popularityEffect )
       × 路線疲勞倍率 × 短程疲勞倍率
淨利 = 運費 − 燃料( loco.fuelPerTrip × 距離係數 × fuelUnitPrice × (1 − 加油站省油%) )
travelDuration = 距離 × distanceToMinutes × (1 + 編組總重 × weightSlowPerUnit) ÷ (loco.speed 正規化) × 疲勞時間倍率 × (1 − 機廠加速%)
```
- Phase 0 成本**先只扣燃料**(薪資/維護 Phase 2 再加)。
- **編組取捨(「加一層」)**:掛越多/越高價 = 運費越高,但越重→在途越慢+油耗略增;掛更多需更強車頭(牽引力 tier 解鎖);節數 ≤ 牽引力 ≤ 月台。→「大編組肥趟 vs 輕快多趟」= 玩家節奏選擇,呼應「抓準次數/時間=最大獲利」。
- **貨種 B 正向特化**(全上緣、零懲罰):チ 泛用×1.0 / ワム 雜貨×1.15 / タキ 油品×1.4 / レム 生鮮×1.6 **+ 準時被取貨 +25%**(原腐壞懲罰改成上緣獎勵)/ コキ 最大載量。
- 常數入 config `balance.freight{baseUnitPrice, distanceRefUnit, weightSlowPerUnit}`;車廂 `valueMult` + `special` 入 `catalogs.cars`。
- 範例:EF210 + 3×コキ(102 載量)送 90 距離的 Tier3 好友 ≈ 102×1.5×(90/30)×1.6 ≈ 735🦷 − 燃料 ~360 ≈ 淨利 ~375🦷/趟(全 config 可調,Phase 0 試玩校準)。

### 15.10 隨機獎勵池（定案 2026-07-15，全上緣、後台全可調）
