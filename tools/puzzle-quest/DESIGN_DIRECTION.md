# 偽文件解謎 — 難度與提示設計規範 v1（正典）

> 起因：CASE-09 昇曜墜樓封測發現「第一階段就能答對正解、提示直接報答案」。
> 這份規範修正**方向**，所有新題目一律照做；交 Codex 驗收時逐條對照本檔 §5 checklist。
> 全程全虛構，勿影射真實個案/公司/人物/判決。

---

## 0. 核心病灶（別再犯）

| 反模式 | CASE-09 實例 | 為什麼壞 |
|---|---|---|
| intro/S1 就把「怪在哪」講白 | intro 直接寫「她死前說**禮拜三要去交東西**」 | 玩家還沒挖就知道有陰謀，後段線索白做 |
| win 關鍵詞在早階段就湊得齊 | 三 core（他殺/滅口/黑心料）憑 S1 直覺就能喊中 | 沒有非到 S4 不可的理由 |
| 沒有誘導性錯誤答案 | 只有「對/不夠」兩態 | 玩家不會被卡住、不會想往下挖 |
| 提示直接報答案 | hint 200 那條＝完整 reveal，core 名詞全講光 | 買一個提示就破台 |
| nudge 幫玩家連線 | 「把那批貨和她禮拜三兜一起看」 | 等於代答 |

---

## 1. 埋深：把 win 鎖在最後階段（最重要）

**鐵律：不到最後階段，玩家湊不齊 win 條件。**

實作三招（可疊加，至少用①②）：

1. **拱心石事實（keystone）＝ 必答 core，且只在 S4/S3+ 文件出現。**
   - 例：致命傷在墜落「之前」形成、放行單筆跡冒簽、門禁刷卡戳破不在場證明。
   - 這個關鍵詞/概念在 S1–S2 的任何文件、poster、留言、intro 裡**一次都不能出現**。
   - solution.core 至少包含 1 個 keystone 概念；玩家早階段答缺它 → 判 partial。

2. **intro / S1 只給「異常感」，不給「異常內容」。**
   - 可以說「大家都當意外，但有幾件事兜不起來」；**不可以**說「她要去檢舉」「她發現黑心料」這類指向動機/對象的句子。
   - S1 三張文件各自看起來正常，只有**跨階段對照**才露餡。

3. **階段門檻（引擎槓桿，建議 Codex 加）**：round.stage < 終章時，即使命中全部 core 也先回
   「你的推論方向對，但手上證據還撐不到定論——案情應該還有下一步。」
   （避免神猜。至少要走到能物理性翻案的那一階段才收 win。）
   - 若引擎暫不做門檻，就靠①的 keystone 硬性擋住。

---

## 2. 誘導：早階段要有「站得住腳的錯誤答案」

- S1–S2 必須讓玩家能拼出一個**有信心、但錯**的結論（自殺/意外/情殺/搶劫/仇殺）。
- 玩家早交這個錯答 → 引擎回「還差關鍵一步」**但不說差在哪**（見 §3 nudge 規則）。
- 真相在 S3–S4 靠一份文件**翻案**（twist）：把稍早的錯方向推翻。
- 紅鯡魚要公平：所有「指向錯答」的線索，事後回看都能被 S4 證物合理解釋，不能是純耍人。

---

## 3. 提示與 nudge：只指路，不報答案

**hint（付費 3 階）規則：**
- 只能講「**看哪張 / 哪兩張對照 / 時間或簽名兜不兜得上 / 換個角度問電話**」。
- **禁止出現任何 core 關鍵名詞**（兇手手法、動機名詞、對象批號/物品名）。
- 分級：50 幣＝指方向（「注意時間軸」）、100 幣＝指要對照的具體兩份文件、200 幣＝點出「有一處對不上」但**不說對上後是什麼**。
- 反例（不可）：「她驗出黑心原料要去檢舉被滅口」。
- 正例（可）：「相驗那張的**時間**、跟門禁刷卡的**時間**，擺一起看順不順。」

**partial / corePartial nudge 規則：**
- 只確認「方向對」＋「還缺一塊」，**不明說缺哪塊**。
- 反例：「把那批貨和她禮拜三兜一起看」。
- 正例：「你講的方向沒錯，但少了能『把意外兩個字劃掉』的那份東西——再往下一步。」

**電話 AI（海龜湯 NPC）規則：**
- 是**劇情線索**不是提示：只答案情事實、不推理不點名（既有 hangupOnAsk/意圖 gate 已符合）。
- 新增要求：NPC 知道的事實**不得涵蓋 keystone**（keystone 只能靠 S4 文件自己看出來），避免用電話跳過最後階段。
- **保底引導（機制已內建，commit cdc80b8）**：玩家連問 3 次都沒問到新線索，第 4、5 次會自動追加一句溫和方向。引導文**優先取該題 `npc.fallbackNudges`**（陣列，建議 2 條：第 4 次一條、第 5 次再明一點點），沒定義才用通用句。**出新題務必寫 `npc.fallbackNudges`**：只給「換個角度／把哪幾張擺一起對」的指路級提示，**同樣不得出現 core／keystone 名詞**（比照 hint/nudge 規範）。判定「有無新線索」靠 NPC 自標 `[[NEW]]/[[SAME]]`（系統自動剝除、玩家看不到）。

## 4. 各階段資訊配比（模板）

| 階段 | 玩家該得到 | 不可出現 |
|---|---|---|
| S1 | 表面故事（意外/自殺）＋ 2–3 個「說不上來哪裡怪」的異常；一個可早交的**錯方向** | 動機名詞、對象名詞、keystone |
| S2 | 死者生前有衝突/異狀，但**衝突內容仍模糊**（可能有好幾種解讀） | 明確動機、keystone |
| S3 | 收斂「對象是什麼」；電話解鎖（只給事實）；仍不知手法/是誰翻案 | keystone（留給 S4） |
| S4 | **物理翻案證物**：把意外→他殺的鐵證 ＋ keystone；此時三 core 才湊得齊 | — |

## 5. 驗收 Checklist（交 Codex／自審都用這張）

- [ ] **早階段測試**：只用 S1（＋S2）的資訊，能不能湊齊 solution.core？→ **必須「不能」**。
- [ ] keystone 概念的所有同義詞，`grep` 過 S1–S2 全部文件文案／intro／poster／留言 → **零命中**。
- [ ] S1–S2 存在一個「有信心的錯答」，且引擎對它回 partial 而非 win。
- [ ] 每條 hint 文案**不含**任何 core 關鍵名詞（逐條 grep core.any）。
- [ ] 每條 nudge 不明說「缺哪塊」。
- [ ] 電話 AI 的 system/知情範圍**不含** keystone。
- [ ] 電話 AI 有寫 `npc.fallbackNudges`（保底引導，建議 2 條），且**不含** core／keystone 名詞（同 hint 規範 grep 一次）。
- [ ] S4 有一份能**物理性翻案**的證物；沒它就無法下定論。
- [ ] 全虛構、時間軸一致（沿用 CASE-09 §7 鎖定法：跨階段時間/簽名/傷勢對得上）。
- [ ] （若引擎做了）stage 門檻：未到終章命中 core 仍回「證據不足」。
