## 0. 核心病灶（別再犯）

| 反模式 | CASE-09 實例 | 為什麼壞 |
|---|---|---|
| intro/S1 就把「怪在哪」講白 | intro 直接寫「她死前說**禮拜三要去交東西**」 | 玩家還沒挖就知道有陰謀，後段線索白做 |
| win 關鍵詞在早階段就湊得齊 | 三 core（他殺/滅口/黑心料）憑 S1 直覺就能喊中 | 沒有非到 S4 不可的理由 |
| 沒有誘導性錯誤答案 | 只有「對/不夠」兩態 | 玩家不會被卡住、不會想往下挖 |
| 提示直接報答案 | hint 200 那條＝完整 reveal，core 名詞全講光 | 買一個提示就破台 |
| nudge 幫玩家連線 | 「把那批貨和她禮拜三兜一起看」 | 等於代答 |

## 1. 埋深：把 win 鎖在最後階段（最重要）

**鐵律：不到最後階段，玩家湊不齊 win 條件。**

實作三招（可疊加，至少用①②）：

1. **拱心石事實（keystone）＝ 必答 core，且只在 S4/S3+ 文件出現。**
   - 例：致命傷在墜落「之前」形成、放行單筆跡冒簽、門禁刷卡戳破不在場證明。
   - 這個關鍵詞/概念在 S1–S2 的任何文件、poster、留言、intro 裡**一次都不能出現**。
   - solution.core 至少包含 1 個 keystone 概念；玩家早階段答缺它 → 判 partial。

2. **intro / S1 只給「異常感」，不給「異常內容」。**
   - 可以說「大家都當意外，但有幾件事兜不起來」；**不可以**說「她要去檢舉」「她發現黑心料」這類指向動機/對象的句子。
   - S1 三張文件各自看起來正常，只有**跨階段對照**才露餡。

3. **階段門檻（引擎槓桿，建議 Codex 加）**：round.stage < 終章時，即使命中全部 core 也先回
   「你的推論方向對，但手上證據還撐不到定論——案情應該還有下一步。」
   （避免神猜。至少要走到能物理性翻案的那一階段才收 win。）
   - 若引擎暫不做門檻，就靠①的 keystone 硬性擋住。

---

## 2. 誘導：早階段要有「站得住腳的錯誤答案」

- S1–S2 必須讓玩家能拼出一個**有信心、但錯**的結論（自殺/意外/情殺/搶劫/仇殺）。
- 玩家早交這個錯答 → 引擎回「還差關鍵一步」**但不說差在哪**（見 §3 nudge 規則）。
- 真相在 S3–S4 靠一份文件**翻案**（twist）：把稍早的錯方向推翻。
- 紅鯡魚要公平：所有「指向錯答」的線索，事後回看都能被 S4 證物合理解釋，不能是純耍人。

## 3. 提示與 nudge：只指路，不報答案

**hint（付費 3 階）規則：**
- 只能講「**看哪張 / 哪兩張對照 / 時間或簽名兜不兜得上 / 換個角度問電話**」。
- **禁止出現任何 core 關鍵名詞**（兇手手法、動機名詞、對象批號/物品名）。
- 分級：50 幣＝指方向（「注意時間軸」）、100 幣＝指要對照的具體兩份文件、200 幣＝點出「有一處對不上」但**不說對上後是什麼**。
- 反例（不可）：「她驗出黑心原料要去檢舉被滅口」。
- 正例（可）：「相驗那張的**時間**、跟門禁刷卡的**時間**，擺一起看順不順。」

**partial / corePartial nudge 規則：**
- 只確認「方向對」＋「還缺一塊」，**不明說缺哪塊**。
- 反例：「把那批貨和她禮拜三兜一起看」。
- 正例：「你講的方向沒錯，但少了能『把意外兩個字劃掉』的那份東西——再往下一步。」

**電話 AI（海龜湯 NPC）規則：**
- 是**劇情線索**不是提示：只答案情事實、不推理不點名（既有 hangupOnAsk/意圖 gate 已符合）。
- 新增要求：NPC 知道的事實**不得涵蓋 keystone**（keystone 只能靠 S4 文件自己看出來），避免用電話跳過最後階段。
- **保底引導（機制已內建，commit cdc80b8）**：玩家連問 3 次都沒問到新線索，第 4、5 次會自動追加一句溫和方向。引導文**優先取該題 `npc.fallbackNudges`**（陣列，建議 2 條：第 4 次一條、第 5 次再明一點點），沒定義才用通用句。**出新題務必寫 `npc.fallbackNudges`**：只給「換個角度／把哪幾張擺一起對」的指路級提示，**同樣不得出現 core／keystone 名詞**（比照 hint/nudge 規範）。判定「有無新線索」靠 NPC 自標 `[[NEW]]/[[SAME]]`（系統自動剝除、玩家看不到）。

## 4. 各階段資訊配比（模板）

| 階段 | 玩家該得到 | 不可出現 |

## 6. 線索媒介與語音（CASE-11 玩後要求，CASE-12 起適用）
- **線索形式要多樣**，別再集中在「LINE 截圖 ＋ 文件/報告卡」。每案讓「截圖類 / 文件類 / 照片(採證照)類 / 其他類」都各有幾張（其他＝語音、手寫便條、收據、行事曆、瀏覽紀錄、地圖、監視截圖、實物特寫…）。
- **每題至少 3 項語音線索**（使用者硬性要求）。使用者用自己的變聲器錄好 → 我用 `tools/bin/ffmpeg` 微調（電話帶通/降噪/音量/加效果）→ 交付。管線與配方見記憶 `reference_audio_clue_pipeline.md`。語音類型例：答錄機留言、行車記錄器收音、監聽片段、匿名報案電話、對話錄音、通聯。
  - 進：使用者貼 Discord **訊息連結** → 我用 bot token（SSM `/sml/discord-bot/token`）打 API 抓附件下載。出：bot POST multipart 發回頻道試聽。
  - 🔴**遊戲內交付前提**：panel 目前只附圖片；語音要在遊戲裡當線索，需先在 `PuzzleQuest.js` panel 加每階段 `audio` 欄位 → push `AttachmentBuilder`（音訊不設 embed image）。此引擎小改動出 CASE-12 時一併做（甜甜重啟生效）。
  - 語音線索**同樣守埋深/提示規範**：早階段語音不得含 keystone/core 名詞；keystone 只在 S4 相關素材。

## 6b. 偵探筆記本 / 證物加工（CASE-13 起適用；引擎已上線）
玩家查案時每階段面板有「📓 存證」鈕：挑**一張**當前階段的照片線索存進私人筆記本，**每案上限 3 格**；「📖 翻筆記」隨時 ephemeral 私下翻閱。CASE-12 玩後回饋（線索就地更新難回翻，但這是刻意設計）→ 用此機制補救，且**存進去的照片會「變化」**（比面板更多/更清楚），讓筆記本贏過純截圖、變成有取捨的策略。
- **引擎位置**：`sweetbot-next/model/PuzzleQuest.js`（按鈕/select/handler）＋ `model/puzzleEvidence.js`（sharp 證物加工）。純本機 sharp、**不燒 LLM、不加成本**。證物版依線索決定性 → 快取在 `data/puzzle-assets/_note/`（不進 git、重啟自動重生）。
- **作者要做的（選用、極輕量）**：在該題 JSON 加一個 `noteMeta`，key＝線索**檔名**，值可含：
  - `focus: [x, y, w, h]`（0~1 比例）→ 存進去時**放大該關鍵區域**成清晰特寫（放大鏡感）。這是「存對線索才看得清細節」的策略核心，**優先給拱心石/含小字或小物證的線索**標。
  - `caption: "一句話"`（中文 OK，走 Discord embed 顯示，不燒進圖）→ 存證卡上的說明。
  - `reveal: "另一張檔名.png"`（進階、選用）→ 存進去才**顯真相**：改用一張暗藏細節的強化圖（浮現隱形字/UV/圈出異常）。要另做圖，留給少數關鍵線索。
  - 範例：`"noteMeta": { "c4-tox.png": { "focus":[0.12,0.55,0.5,0.2], "caption":"毒物報告·乙二醇代謝物" } }`
- **沒標 `noteMeta` 的線索**：仍可存，只是不放大——一律套「證物版」風格（泛黃＋EVIDENCE 章＋CASE FILE 邊框），存了就跟截圖不一樣。
- **埋深規範照舊**：`focus`/`reveal` 若會放大/顯露 keystone，等同該素材本身——**只有 S4 線索可以指向 keystone**；早階段線索的 focus 別放大出 core/keystone 級資訊。

## 8. CASE-13 起新增機制（現行機制，出新題一律照這走）
> 這一輪封測討論拍板三項機制變更＋一個多層真相寫法。**下列為出題現行標準**；標「需引擎改動」者，出 CASE-13 時一併做（sweetbot commit＋離峰重啟才生效），沒做前用「作者側資料/文案」替代。

### 8.1 靜默階段獎勵（推進即給獎，不洗版、不通知）
- **改法**：玩家每推進到新階段，**當下就發一小筆牙齒獎勵**（例：每階段 +30~50🦷，實際數值後台可調當煞車），而非只有最後破案才發。目的＝讓「往下挖」本身有正回饋，接住 CASE-12「S2–S4 變驗證、沒動力挖」的問題。
- **靜默**：獎勵**沿用 CASE-11 的靜默推進**（commit d5e1a10）——**原地更新 embed、不發新訊息、不 @、不洗版**；牙齒入帳走既有 `givePoint`，只在面板角落淡淡標「＋N🦷（推進獎勵）」即可，不另發通知。
- **與破案獎的關係**：階段獎是「零錢」、破案 `reward.base` 仍是大頭；破案總獎**不扣**已領的階段獎（階段獎是額外的，不是預支）。作者在 JSON 用 `reward.stageBonus`（單階段額度，缺省給預設）表示。
- **需引擎改動**：現行引擎只在 solve 給獎（`rewardWithoutBonus`），推進不給。要在 `advanceActiveCase`/推進路徑加「首次進入某階段 → givePoint(stageBonus) + 冪等記已領」。冪等鍵＝`roundId+userId+stage`，**同一人重看同階段不重複領**。

### 8.2 取消「答錯 2 次出局」（只留答題費當煞車，不淘汰玩家）
- **拍板**：**不設任何「答錯 N 次就鎖定/淘汰」的硬出局**。玩家想猜幾次都行。
- **唯一煞車＝既有累進答題費**（`answerFeeForAttempt`：第 1 次免費、第 2 次 100🦷、第 3 次 200🦷…線性 +100）——用「越猜越貴」溫和抑制亂槍打鳥，而不是把人踢出局。
- **理由**：解謎重體驗、封閉社群小圈子，硬出局＝挫折又流失；付費遞增已足夠讓人「先想清楚再送答」。**任何新題/後台都不要再加出局或鎖答上限。**
- 引擎現況：本來就沒有出局邏輯（只有累進費）→ 此條是**明文禁止未來加**，非要改碼。

### 8.3 電話 AI：熱門話題「暢談」、離題「冷淡」
> 這是 CASE-11「NPC 大部分問不出線索、幫助不大」的根治升級，取代「知道清單寫厚」的模糊要求，改成**可操作的兩檔語氣**。
- **熱門話題（hot topics）＝作者為該案指定的一組核心可談主題**（死者作息/人際/當天細節/各關係人的小事/現場氛圍…**凡不含 keystone 者皆可**）。玩家問到熱門話題 → NPC **暢談**：主動多吐 1–2 句有味道的背景，做到「**每問必有點東西**」，有人味不乾答。
- **離題（off-topic）＝與案情無關的閒扯、或試圖套 keystone/要 NPC 指認** → NPC **冷淡**：簡短帶過、不擴充、守住 gate（keystone 只能靠 S4 文件自己看出，NPC 不碰）。
- **作者要做的**：在該題 JSON 的 `npc` 加 `hotTopics`（陣列，每項＝主題關鍵詞＋NPC 可暢談的 1–3 條事實，**逐條不得含 core/keystone**，比照 hint 規範 grep 一次）。`fallbackNudges`（§3 保底引導）續寫。門檻沿用 `[[NEW]]/[[SAME]]` 自標。
- **需引擎改動**：`PuzzlePhoneNPC` system prompt 要吃 `hotTopics` 並灌「命中熱門→暢談、離題→冷淡」的語氣指令；沒做前先靠把 `npc.system`「知道」清單寫厚（CASE-11 既有做法）替代。

### 8.4 多層真相寫法（紅鯡魚人物 + 更深一層動機；CASE-13 示範）
- **鼓勵「表層嫌犯是紅鯡魚、真兇在更深一層」**的雙層結構：早階段線索把矛頭導向一個站得住腳的錯人（見 §2 誘導），S3–S4 才翻案到真兇，且真兇的**動機在更底層**（如利益/掩蓋另一樁罪）。
- **CASE-13 採用**（作者定案、細節見 `archive/LESSONS.md` §2 CASE-13 規劃條）：以某關係人的**筆記本當紅鯡魚**把人帶偏 → 真兇是另一人 → 最底層動機＝**贗品／洗錢**這條經濟犯罪線。keystone 仍鎖 S4、照 §1 埋深。
- 規範：**紅鯡魚要公平**（§2 末條）——所有指向錯人的線索，事後回看都能被 S4 證物合理解釋，不能純耍人；洗錢/贗品這種「深層動機」的決定性文件也算 keystone 級，**只能在 S4 出現**。
