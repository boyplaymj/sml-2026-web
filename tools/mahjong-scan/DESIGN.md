# 🀄 手機掃描牌 → 判斷聽什麼（mahjong-scan）— 設計冊

> **一句話**：手機相機拍下自己整手牌 → 端上辨識出 14/16 張牌 → 丟給既有的聽牌核心 → 秒回「聽哪些牌」。
>
> 定稿日：2026-07-18　狀態：**v0.1 設計，待交 Codex**　對口＝gameboy
>
> 產品歸屬：先做 **standalone demo**（仿 `mahjong-voice-tai` 語音判台模式，掛圖床），兩雀原始碼到手後再嵌入兩雀 App。

| 部件 | 狀態 | 來源 |

## 1. 目標與範圍

**做**
- 手機瀏覽器開相機（或選相簿照片）→ 框出/自動偵測整手牌 → 顯示「聽 ⚀⚁⚂…」。
- 辨識信心低時，讓使用者**點一下手動校正**（少數牌，不是全部重輸）。
- 全程端上（on-device）推論，離線可用、$0、不外洩牌面。

**不做（v1 範圍外，列 Phase 2/風險）**
- 台數/番種計算（聽牌核心本就只判結構、免役，見 SPEC §6）。
- 槓（4 張面子）與複雜副露的張數推算 → v1 假設門清或平放的碰/吃。
- 即時連續影格追蹤（v1 是「拍一張 → 出結果」，非串流）。

## 2. 系統架構 / 資料流

```
[手機相機/相簿]
      │  一張照片（整手牌平放或持牌貼近拍）
      ▼
[① 端上偵測]  YOLO(tfjs/onnx, WebGL/WASM) → 一堆 bbox + 類別標籤 + 信心
      │  e.g. ["3m"@.98, "3m"@.95, "5p"@.71, "e"@.99, ...]
      ▼
[② 標籤→牌id 映射 + 花牌分流]  §5 對照表
      │  概念手牌 ids = [3,3,25,101,...]（升冪）；花牌另存不進聽牌
      ▼
[③ 校正 UI]  低信心/張數不符 → 提示使用者點選修正
      │  最終確認的 ids（張數須 3k+1）
      ▼
[④ 聽牌核心]  MahjongHand.waitingTiles(ids)  ← 既有純函式，零改動
      │  → [4, 7]（聽四萬、七萬）
      ▼
[⑤ 顯示]  以牌面圖示呈現「聽：4m 7m」，可展開每張的胡型
```

**關鍵原則**：① 只負責「這是什麼牌」，④ 只負責「聽什麼」，中間靠純資料（id 陣列）解耦。辨識引擎日後升級（換模型/雲端）完全不動聽牌核心。

## 3. 辨識引擎：重用直播模型 → 匯出端上

### 3.1 為什麼不丟 LLM 讀？
多模態 LLM 讀一排 14 張長得極像的牌（三索/四索、六筒/八筒、花牌兩組相反配色）**易誤判、會漏數、且每次燒 token**。固定 42 類的辨識，物件偵測器（YOLO）才是本行，且端上跑 **$0**。

### 3.2 重用路徑
1. 直播 session 的 Roboflow 專案已標 42 類（`project_mahjong_vision`）。**同一份資料集/權重**匯出成瀏覽器可跑格式：
   - Roboflow → **TensorFlow.js** 或 **ONNX**（走 `onnxruntime-web`，WASM/WebGL）。
2. demo 頁載入該模型檔（放圖床同源），首次載入後快取，之後離線可用。

### 3.3 唯一的「小工」：視角落差
- 直播模型＝**桌面俯視角**訓練；手機掃描＝**手拿一排、貼近、斜角**。域不同 → 直接套可能掉準。
- **解法（非重練）**：補拍 **20–50 張手機實拍**（不同光線/角度/牌數）丟進同一 Roboflow 專案，標一標、微調再匯出一版。標籤沿用原 42 類，零新類別。
- 補拍計畫寫進 Phase 1 交付項。

## 4. 手牌張數契約（台灣 16 張制）

聽牌核心的契約（SPEC §4）：
- **聽牌 = 3k+1 張**：`1 / 4 / 7 / 10 / 13 / 16`。門清等一張時手上正好 **16 張** = 3k+1 ✓。
- `waitingTiles(ids)`：張數非 3k+1 → 回 `[]`。

**副露處理（v1）**：把「手上暗牌 + 已平放的碰/吃面子」全部一起掃進來，總張數維持 3k+1，結構照樣成立（核心不分暗明，只看結構）。
**槓（4 張面子）＝ v1 已知限制**：槓用掉牌牆額外一張，張數會多 1 → 破契約。v1 先不支援；校正 UI 若偵測到某牌 4 張且結構異常 → 提示「含槓暫不支援」。Phase 2 補槓旗標。

## 5. 模型標籤 → 牌 id 對照表（**接線關鍵**）

直播模型輸出的是字串標籤，聽牌核心吃整數 id（SPEC §3 編碼）。映射如下，實作為一張常數表：

| 類別 | 模型標籤 | → 牌 id | 備註 |
|---|---|---|---|
| 萬 1–9 | `1m`…`9m` | `1`…`9` | 數字即 id |
| 索 1–9 | `1s`…`9s` | `11`…`19` | 數字+10 |
| 筒 1–9 | `1p`…`9p` | `21`…`29` | 數字+20 |
| 東 | `e` | `101` | 風牌，單字母 |
| 南 | `s` | `201` | ⚠️ 單 `s`=南；`1s`=索，**靠有無數字前綴區分** |
| 西 | `w` | `301` | |
| 北 | `n` | `401` | |
| 中 | `r` | `501` | 三元（紅中） |
| 發 | `g` | `601` | 三元（青發） |
| 白 | `b` | `701` | 三元（白板） |
| 季節花 春夏秋冬 | `1z`…`4z` | **（不進聽牌）** | 花牌另存，見 §5.1 |
| 植物花 梅蘭菊竹 | `1y`…`4y` | **（不進聽牌）** | 同上 |

```js
// tools/mahjong-scan/labelToId.js（提議）
const SUIT = { m: 0, s: 10, p: 20 };
const HONOR = { e:101, s:201, w:301, n:401, r:501, g:601, b:701 };
const FLOWERS = new Set(['1z','2z','3z','4z','1y','2y','3y','4y']);

function labelToId (label) {
  if (FLOWERS.has(label)) return null;                 // 花牌：不進手牌
  if (/^[1-9][msp]$/.test(label))                      // 數字牌
    return SUIT[label[1]] + Number(label[0]);
  if (HONOR[label] != null) return HONOR[label];       // 字牌（單字母）
  throw new Error('unknown tile label: ' + label);
}
```

### 5.1 花牌分流
花牌（`*z`/`*y`）在台灣麻將是**外露補花、不參與胡型**。辨識到就從概念手牌剔除、單獨列在 UI（「你的花：春、梅…」），**絕不能**混進 `waitingTiles` 的 ids，否則張數與結構全錯。

## 6. UI / UX（demo 頁）

1. **拍照/選圖**：大按鈕開相機 or 相簿。建議引導「牌面朝上、平放成一排、盡量填滿畫面」。
2. **偵測結果條**：底部橫向排出辨識到的牌面小圖，每張標信心；花牌獨立一欄。
3. **校正**：
   - 低信心（<閾值，如 .6）的牌**閃黃框**，點一下彈 42 類選盤改正。
   - 張數不是 3k+1 → 頂部紅字提示「目前 N 張，聽牌需 16（或 13/10…）張，請補拍或手動增刪」。
4. **結果**：張數對且結構有解 → 大字「**聽：4m 7m**」以牌面圖示呈現；可展開看每張聽牌對應的完整胡型（用 `isHu` 驗證）。
   - 無聽（`waitingTiles` 空）→「詐胡/未聽，還差牌」。
5. **零後端**：整頁純前端，模型 + `MahjongHand.js` + `labelToId.js` 都在瀏覽器。

（前端打磨可另跑 `impeccable` skill；i18n 圖片不內嵌文字，牌面用圖示、說明走文字。）

## 10. 待決策 / 風險

1. **模型匯出格式**：tfjs vs onnxruntime-web —— 看直播 session 訓出來的權重與 Roboflow 匯出選項，P1 開頭定。（onnx 通用性高、tfjs 與瀏覽器整合成熟，兩者皆 $0。）
2. **手機視角掉準幅度**：補拍量（20 vs 50 張）取決於實測落差 → P1 先拍一小批 A/B。
3. **一排牌的偵測穩定度**：14–16 張並排、邊緣遮擋、斜牌 → 若整排偵測不穩，退路＝引導使用者「分兩段拍」或「牌距拉開」。
4. **標籤衝突已解**：單 `s`（南）vs `1s`（索）靠「有無數字前綴」區分（§5），實作務必用正則 `^[1-9][msp]$` 而非 `includes('s')`。
5. **槓 / 暗槓**：v1 不支援，P2 補（張數 +1 的旗標處理）。
6. **聽牌核心來源**：以 `tools/mahjong-hand-core/SPEC.md` §5 內嵌實作為權威逐字副本，demo 與 sweetbot 版本必須同源，改動先改 SPEC。
