# 🌱 角色成長引擎 (RPG)

> **狀態**：設計草案 🔶（Claude 提案，待 gameboy 拍板）
> 一句話：**把「真實打牌的軌跡」長成一個看得到、對找咖有意義的角色——RPG 是資歷與經歷，不是戰力、不是排名。**

---

## 0. 定位：RPG ≠ 天梯（先切清楚，否則兩個模組會打架）

foundation §3.4 把可信對局的扇出分兩批。**RPG 落在「立即成立(帳與自身)」那批——不等爭議解除就結算**。這條決定了 RPG 的整個性格：

| | 🌱 RPG 成長（本模組） | 🏆 天梯（[ladder](ladder)） |
|---|---|---|
| 本質 | **資歷／經歷**（打過多少真實的牌） | **實力排名**（打得多強） |
| 方向 | **只增不減**（資歷不倒退） | 可升可掉、賽季軟重置 |
| 結算時機 | **可信對局成立即刻**（含 3/4 帶異議） | **等爭議解除**才餵（擱置） |
| 對找咖的意義 | 「這人是真的常打、不是殭屍帳號/新雷」 | 「這人打得多好」 |
| 付費關係 | 外觀解鎖靠真實進度（見 §4） | **絕不賣分**（pay-to-win 紅線） |

**因為 RPG 不等爭議解除就結算 → 防刷是本模組的命門**（§5），否則假帳互刷就能長出漂亮角色，直接汙染名片可信度。

---

## 1. 名片鐵律套用（foundation §2.3）

角色身上每一個「看得到」的屬性，都必須對「正在決定要不要跟你打牌的人」有意義。純為升級而升級 = 雜訊，砍。本模組三個對外屬性各自回答一個問題：

- **等級** → 「這人夠活躍、夠真實嗎？」（資歷）
- **稱號** → 「這人有什麼可信的招牌？」（成就，綁真實訊號）
- **外觀** → 「這人願意經營自己嗎？」（表現力，純裝飾不影響公平）

---

## 2. 經驗來源（XP）——錨在「真實參與」，不錨在「贏」

**鐵律：XP 獎勵「打了一場真實的牌」，不獎勵「贏」。** 不隨輸贏放大 → 同時擋 pay-to-win 觀感與大分互刷。

| XP 來源 | 給法 | 為何這樣 |
|---|---|---|
| **完成一場可信對局** | 每場**固定基礎 XP**（不看名次/輸贏） | 資歷=參與真實社交打牌，跟強弱無關 |
| 首次與新牌搭同桌 | 小額一次性加成 | 獎勵「擴大真實牌搭圈」，對社群健康有利 |
| 連續活躍（週 streak） | 溫和加成、斷了只歸零不倒扣 | 鼓勵回訪、但不逼肝 |
| 完成信譽/場評回填 | 小額 | 獎勵讓生態資料更完整的行為 |

- **可選微調**：名次可帶**極輕微**係數（1st 略高於 4th），但幅度小到不足以誘導做局。**Claude 傾向純固定 XP 起步**（最防刷、最好解釋），微調留 v2 再評估。
- **solo 個人帳本永不給 XP**（見 §5 不變式）——只有可信對局（≥3 同意、已零和驗）才長角色。

---

## 3. 等級與稱號

### 3.1 等級 = 累積資歷里程碑（只增不減）
- 等級由 `totalXp` 換算，**對數式曲線、無硬上限**（資歷永遠能長，但邊際遞減 → 高等不氾濫）。
- **等級明確不是實力**（實力看天梯段位）、**不是外觀等級**。名片上它只證明「真實活躍度」。
- **近期活躍度另軌顯示**（`lastMatchAt` / 近 30 天場次），避免「高等殭屍帳號」誤導找咖的人——等級答「歷史夠不夠真」，活躍度答「現在還在不在」。

### 3.2 稱號 = 成就式，每個都對找咖有意義
純娛樂／無訊號的稱號不做（名片鐵律）。初始四類，都綁**可驗的真實訊號**：

| 類別 | 例 | 綁的訊號 |
|---|---|---|
| **資歷** | 百局老手 · 千局雀鬼 | 可信對局場次 |
| **信任**（接 [reputation](reputation)） | 零放鴿 · 準時之星 · 好牌品 | 👍評價/準時率門檻 |
| **場**（接 [parlor](parlor)） | ○○館常客 · 開場百次場主 | 到場/開場統計 |
| **競技**（接 [ladder](ladder)） | 上季鑽石 | 賽季結算快照 |

- 玩家**選一個當名片主稱號**，其餘進收藏。
- 違規/反托標記者的相關對局**不計入稱號門檻**（接 [trust-safety](trust-safety) §58）。

## 4. 外觀系統（三層，接 business §2.4 + §3 護欄）

「外觀非照片」（foundation §2.4）：用遊戲角色外觀養收集與變現，同時保護隱私。三層：

1. **預設**：免費基礎捏臉 + 基本組件，人人有。**絕不 paywall 身分**。
2. **可組裝解鎖**：靠**真實對局進度**（賽季通行證軌 / 成就）解鎖組件 → 這是 RPG↔[business](business) 的接點，外觀是「打真牌賺來的」，促留存。
3. **直購/生成**：高階主題用 Bedrock 生成素材，**直購（非 gacha）**。

**護欄（繼承 business §3）**：
- ① 外觀純裝飾，**不影響對局或媒合公平**（不 pay-to-win）。
- ② **不做轉蛋**——只直購/通行證（走賭博灰線的 app，機率型付費是地雷）。
- ③ **認證徽章 ≠ 外觀**：認證=打真牌賺來、系統核發、**不可購買**（business §2.5 護欄）。

---

## 5. 防刷不變式（§ 給 Codex 寫測試——本模組的命門）

因 RPG 立即結算、不等爭議解除，以下為**硬約束**，違反即角色系統失去可信度：

1. **solo 帳本永不給 XP**——XP 只源自 `PK=MATCH#` 的可信對局。
2. **可信對局門檻**：僅 ≥3 同意（已零和驗、加總=0）的對局計 XP；<3 退回個人帳本者不計。
3. **同組遞減**：固定四人反覆對打，**同組合的 XP 逐次遞減**（防固定四人互刷）。
4. **每日/每週 XP 上限**：單人 XP 有日/週 cap（防單日狂灌）。
5. **違規排除**：反托/違規標記的對局，**排除或不計入** XP 與稱號門檻（接 trust-safety）。
6. **等級只增不減**；**外觀不影響對局/媒合公平**；**認證徽章不可買**。
7. **XP 全程可稽核**：每筆 XP 進 `XpLedger`（含 matchId、reason、是否觸 cap），供後台審計與申訴回溯。

## 6. 資料模型草案 🔶

```
CharacterProfile { userId, level, totalXp, verifiedMatches,
                   activeStreakWeeks, lastMatchAt,
                   mainTitle, titles[], appearance{slots...} }
XpLedger  { userId, matchId, deltaXp, reason, dailyCapHit(bool), ts }
           // 防刷稽核流水，PK=USER#<uid> / SK=ts
TitleDef  { titleId, name, category(tenure|trust|venue|ladder), rule }
AppearanceItem { itemId, slot, source(default|unlock|purchase), unlockRule }
```
角色成長吃 foundation §3.4 的「可信對局成立」事件；與天梯**同源不同時**（RPG 立即、天梯擱置）。

## 7. API 草案（沿用 App 命名慣例）🔶
`GET /character-me` · `GET /character/:userId`（名片公開視圖）· `POST /character-apply-xp`（對局成立內部觸發）· `GET /character-titles` · `POST /character-set-title` · `POST /character-appearance`（存/換裝）

## 8. 後台管理面板（v2）
XP 表與 cap 調參、稱號門檻 CRUD、外觀組件上下架/解鎖規則、XpLedger 稽核查詢、手動修正（作弊扣經歷/收回不當稱號）、外觀生成用量卡（若上 Bedrock）。

## 💰 成本控管（遵循 tools/COST_CONTROL.md）

- **MVP（無 AI）**：成本來源僅 DDB 新表 `character-profile` / `character-xp-ledger` / `title-def` / `appearance-item`，PAY_PER_REQUEST，讀多寫少、量級小（預估 < $2/月）。無 LLM／無付費 API → 免帳本封頂。
- **外觀生成**：走 Bedrock 圖像（Stability SD3.5, us-west-2），**按張、直購時才生**、產物快取進圖床 → 可控、非迴圈燒錢。
- **若日後加「AI 教練/戰報」**（分析打法給建議）→ 走 Bedrock `global.anthropic.claude-opus-4-8` 無 key，並補齊四件套（帳本表 + 月封頂 cap + 後台用量卡 + kill switch）。
