## 0. 定位一句話

每位玩家在**全服共用的地圖**上開一間麻將館,靠**選點、經營、宣傳、對抗**在持續運轉的世界裡求生,撐越久、開越大越強。

**四大決策(已定案):**
- 平台 → 甜甜 Discord bot MVP(重用圖床/emoji拼圖/牙齒經濟)
- 多人 → **即時**:Lv1 連續世界 + Lv2 即時對抗(非 Lv3 RTS)
- 節奏 → 連續 tick(掛機式),非每日結算
- 貨幣 → **單一牙齒🦷**(本質非賭博,全程牙齒;防通膨靠強力消耗口自然回收)
- 內容分級 → 非賭博,無分級疑慮;事件/臨檢走輕鬆娛樂化

## 1. 核心迴圈

```
選點開館 → 拉客(客流分食) → 服務(翻桌/客單價) → 收益
   → 再投資(擴桌/升級/宣傳) → 應付對抗與隨機意外 → 求生存活
```

存活越久 → 解鎖更大牌桌/更好地段/更強宣傳 → 正回饋。

## 2. 世界與客源(Lv1 連續世界)

### 2.1 地圖 → **完整規格 `CODEX_SPEC_worldmap.md`**(選點圖引擎/天王溪 feature/天候 overlay/戰況板;≠ 店面平面圖)
- 地圖 = 數個**行政區**(MVP 先 3 區),之後可擴。
- 🔴 **行政區身分待統一**(worldmap §9):CONTENT 6 區(便利商店味)vs §2.1.1 天候 3 區(地勢味)vs 引擎舊 3 區 id 三套並存,MVP 前需拍板收斂(建議以 §2.1.1 正典為準收斂 3 區,動到已上線 config 故待使用者選)。
- 每區屬性:
  - `baseFlow`:每小時基準客流量(客人單位/hr)
  - `clientMix`:客層分布(**10 客群**:散客/雀友/大戶/觀光客/學生/👴高齡/👩‍👧媽媽/🚸翹課學生/🎲游擊中年人/🌱麻將新手 佔比;canonical key=`casual/regular/whale/tourist/student/elderly/mama/truant/roamer/novice`;皆正式客群、都進 clientMix)
  - `rentLevel`:租金水位(熱區貴)
  - `riskLevel`:治安/臨檢風險
  - `heat`:區域熱度(會隨玩家聚集上升 → 帶動租金上漲,逼成長)
  - **`terrain` 地勢屬性(固定不變,選址策略深度來源,§17 天氣共用)**:`floodRisk` 淹水風險 / `windExposure` 風勢暴露 / `heatIsland` 熱島 —— 天氣「共享鋒面」打全服,但同一場雨落在不同區後果不同(見 §2.1.1、§17.1a)。

### 2.1.1 天王里正典地圖(**已定案 2026-07-16:自建虛構地圖,不對應玩家真實 GPS**)
> **為何要自建**:天王里是社群**既有的虛構里**(選舉系統同一個里),不是玩家所在的真實地區。**天氣分兩種顆粒度**(2026-07-16 使用者定調):①**極端(颱風/全台大雨)= 真實層觸發、全服一致**(這種本就是全台一起的大範圍鋒面,見 §17.1a);②**日常(晴~大雨)= 各區遊戲內隨機**(台灣午後雷陣雨本就局部,各區不同反而真實、選址意義最大)。**模擬層自己不擲颱風級 → 永遠不會出現「A 區隨機颱風、B 區晴天」的穿幫**(全台一致的極端只從現實來)。建圖時仍把各區相對位置與地勢一次畫死成正典(共用地圖已定 §13),讓玩家看得懂「為什麼我這區先淹」——是可預測的地勢,不是 RNG。

- **內部地理(相對位置固定,一張正典圖,全服共用)** —— 天王里以一條 **天王溪** 貫穿,MVP 3 區各占一個地勢極端軸,讓選址教學一眼看懂:

| 行政區 | 相對位置 | `floodRisk` | `windExposure` | `heatIsland` | 租金 | 玩點 |
|---|---|---|---|---|---|---|
| 🌊 **溪畔區** | 西南・臨天王溪低窪 | **高** | 低 | 中 | 低 | 便宜但梅雨/豪雨先淹 → 逼買抽水機 |
| 🏙 **舊城市場區** | 中央・密集老街 | 中(舊排水) | 低 | **高** | 高 | 最旺最貴,酷暑冷氣電費最兇 |
| 🌾 **海線開闊區** | 東・臨海田邊空曠 | 中 | **高** | 低 | 中 | 颱風招牌掉落/停電風險最高 → 逼防風鐵捲門+發電機 |

- **擴充區(留 M2/連鎖)**:🏔 高地重劃新市鎮(北・高地,flood 低/wind 中/heat 低,租金最高——安全又新);之後可再擴。
- **一句話**:日常各區隨機、極端(颱風/全台大雨)全服一致從現實來;影響看各區地勢,地勢是**建圖時定死的正典**,選址 = 選這間店「永久的天氣體質」。

### 2.2 連續客流 + 惰性結算
- 世界**持續在跑**:每區每單位時間產生客流,按同區各館的**吸引力分數**比例分食。
- **惰性結算(lazy tick)**:不需要每秒跑全服。玩家一互動,就用 `now - lastSettledAt` 補算這段期間的客流/營收/損耗/事件。省算力、天然「掛機」。
- **世界 tick(全服低頻)**:每 5–15 分一次,只做:更新地圖看板、推進區域熱度/租金波動、擲全服隨機事件、對手 NPC 行為。

### 2.4 時段尖峰客流(v0.2 借入《便利商店》)
- 一日分時段,客流量乘上**時段係數**(離峰↓、尖峰↑)。麻將館典型:
  - 早(離峰)、午(小尖峰,退休雀友)、**晚間~深夜(主尖峰,上班族下班/夜貓)**。
  - 各區 `clientMix` 也可隨時段變(白天雀友多、深夜大戶多)。
- 惰性結算要**按遊戲內時間積分**經過的各時段(不是只用平均):`revenue = ∫ flow(t)·capture dt`。
- 玩家用**營業時間 + 排班**去對準尖峰:尖峰沒開/沒排人 = 把錢讓給對手。
- 時段係數表放 `balance` 設定,後台可調。

### 2.3 吸引力分數(核心公式雛形)
```
attractiveness = w1·聲譽 + w2·環境 + w3·餐飲 + w4·宣傳熱度
               + w5·價格競爭力 + w6·荷官品質 + w7·牌桌品質
```
- 權重 `w` **依客層而異**:
  - 大戶 → 重 VIP包廂/環境/荷官
  - 散客 → 重 價格/便利
  - 雀友 → 重 桌況/常客氛圍/餐飲
  - 觀光客 → 重 話題性/宣傳熱度
- 同區客流按各館 attractiveness 做比例(softmax)分配 → 你強就搶更多,對手削價你就掉客。

## 3. 經營維度(玩家能調的旋鈕)— v0.2 借入《便利商店》深度

| 維度 | 內容 | 影響 |
|---|---|---|
| **設備採購** | 麻將桌(普通/電動/VIP)、椅子、冰箱、空調、飲水機、廁所、監視器、按摩椅… | 容客量、環境值、客層、效率;部分**解鎖服務/客群**(見 §3.3) |
| **服務項目** | 免費宵夜、代客泊車、接送、線上訂桌、會員卡、洗牌機 | 特定客層吸引力↑、停留/回頭率↑ |
| **進貨/庫存** | 飲料、香菸、宵夜食材、耗材(牌尺/籌碼)… 主動叫貨 | 有**庫存與損耗/過期**,缺貨趕客;賣出賺價差(見 §3.4) |
| **人力/排班** | 門市人員(荷官/服務生/廚房/保全/櫃檯),各有等級/薪資/士氣 | 效率、聲譽、防踢館;**要排進時段班表**(見 §3.5) |
| **營業時間** | 設定每日開店時段(可 24h 或分時段) | 覆蓋哪些**客流尖峰**(§2.4) vs 人事/水電成本 |
| **價格** | 檯費 / 抽水率 | 高賺多但趕客(價格競爭力↓) |
| **環境/清潔** | 裝潢等級、清潔度(會隨營業下滑要維護) | 聲譽、客層升級 |

### 3.3 設備採購(一次性資產 + 維護費)
- **一次性採購**(花牙齒),長期持有;部分有**每 tick 維護費/水電**。
- 效果面向:`容客量`(桌/椅)、`環境值`、`客層解鎖`(VIP包廂→大戶)、`效率`(洗牌機→翻桌率、監視器→防踢館)、`服務解鎖`(冰箱→賣冷飲、廚房→宵夜)。
- 有**折舊/故障**隨機事件掛鉤(冷氣壞、桌子壞)。

### 3.4 進貨 / 庫存(便利商店式核心迴圈)
- 消耗品(consumables)要**主動進貨**:成本、售價、**保存期限/損耗率**、佔用空間(受冰箱/倉儲上限)。
- 客人消費會**扣庫存**;缺貨 → 該筆餐飲收入喪失 + 小幅聲譽/停留↓。
- 過期 → 損耗計入支出。**進貨太多壓資金、太少缺貨**,形成決策張力。

### 3.5 門市人員 / 排班 / 營業時間
- 招募門市人員(職務:荷官/服務生/廚房/保全/櫃檯),各有**等級、薪資、士氣**(裝備/福利提升士氣→效率)。
- **排班**:把人員指派到營業時段;某時段沒排足人 → 服務力不足、翻桌率/聲譽↓、客人流失。
- **營業時間**:決定覆蓋哪些客流尖峰;開越久客流覆蓋越廣但薪資/水電越重、清潔下滑越快。
- 薪資只在**排班時段**計費(對齊真實)。

## 4. 收益與支出(結算)

```
收入 = 檯費/抽水 + 餐飲 + 會員費 + 活動收入
支出 = 租金 + 薪資 + 進貨(餐飲/耗材) + 水電 + 宣傳費 + 罰款/賠償
```
- 現金流長期為負且儲備見底 → 進入**危機狀態** → 連續赤字達門檻 → **倒店**。
- 倒店:保留玩家等級與牙齒,館體歸零,可花牙齒重開(復活/新址)。

## 5. 宣傳系統

- 手段:傳單 / 社群貼文 / 開幕活動 / 小型比賽 / **名人站台(SML 選手彩蛋)**。

- 成本:一般宣傳便宜、高級宣傳(名人/跨區曝光)貴,全付牙齒🦷。

## 6. 即時對抗(Lv2 — 立即結算 + 通知對手)

| 指令 | 效果 | 對手反制 |
|---|---|---|
| 削價戰 | 降自己抽水搶同區客,傷利潤 | 跟進削價 / 走高端差異化 |
| 踢館 | 派人鬧場,降對手聲譽(有成本/風險) | 保全等級可擋、可反報警 |
| 挖角 | 高薪挖對手明星荷官 | 加薪留人 / 提前簽約 |
| 謠言/商業間諜(灰色,後期) | 抹黑降聲譽 | 公關危機處理 |
| 結盟/連鎖(正向) | 共享宣傳、聯合比賽、集客 | — |

> 對抗是「**事件級即時**」:幾秒內對手收到甜甜通知並能回招。非畫面級即時。

## 7. 隨機事件表(便利商店式二選一)

每事件 = 觸發條件 + 情境 + 2 選項分支 + 結果。範例池:
- 警察臨檢(risk 高的區更常見)
- VIP 大戶要包場(接=大收入但趕散客 / 拒=保住常客)
- 打手/詐賭上門
- 跳電 / 冷氣壞 / 食安事件
- 節慶爆客(春節/中秋)
- 黑道收保護費
- 媒體來採訪(聲譽波動大)
- 明星荷官被挖 / 鬧離職
- 樂透中獎客進來灑錢

## 8. 求生與難度曲線

- 租金隨區域熱度/時間**上漲** → 逼玩家持續成長。
- **對手 NPC**:玩家不足的區用 NPC 館填充,保證每區都有對抗張力。
- 難度靠「租金爬升 + 對手變強 + 事件變凶」推進,而非硬數值牆。

## 9. 貨幣:單一牙齒🦷

- 全程只用**牙齒**(本質非賭博,館內輸贏=牙齒經濟流動;無真金錢、無分級問題)。
- 開館入場、經營投資、復活、宣傳、對抗全用牙齒;串既有牙齒經濟,玩家不用學兩套幣。
- **防通膨**靠強力消耗口自然回收:租金(隨區域熱度上漲)、薪資、水電、升級、罰款。麻將館經營天生大 sink,收支自平衡。
- (若日後要更深的經濟博弈,再評估加「不可提現的館內籌碼」內層;MVP 不做。)

## 13. MVP 切期(每塊 <25 分工作量,分階段驗)

- **Phase 0**:資料表 + 單一指令面板 + 開館 + 惰性營收 → 單人開館、掛機賺牙齒、提款、最小倒店。(先驗核心手感,**不含**設備/庫存/排班)
- **Phase 1「經營深度」(便利商店味)**:設備採購 + 服務 + 進貨/庫存/損耗 + 門市人員/排班 + 營業時間 + 時段尖峰客流。後台型錄新增 equipment/services、balance 加 timeBands。
- **Phase 2**:地圖/客源 attractiveness 分食 + 地圖看板頻道 + 對手 NPC。
- **Phase 3**:宣傳系統 + 隨機事件表(含設備故障/缺貨/奧客)。
- **Phase 4**:即時對抗(削價/踢館/挖角)+ 通知。
- **Phase 5**:平衡調校 + 牙齒經濟後台串接。

> 註:設備/庫存/排班(Phase 1)刻意排在核心手感(Phase 0)之後、分食對抗之前——先讓單館經營夠厚,再加多人。

## 17. 氣候系統(設計 v0.1,2026-07-16 使用者點名)

核心 = 天氣同時「**軟性調客流**」+「**硬性擲極端事件**」,而**招牌機制 = 行政放假 vs 實際天象的落差**(使用者點名)。

### 17.1 混合天氣模型(**已定案 2026-07-16:真實 + 模擬雙層**)
天氣 = **真實層(宏觀共享)+ 模擬層(微觀各區)** 疊加,拿真實天氣的臨場感、又不吃它的坑:

- **真實層(宏觀 / 稀有 / 全服共享)** — 只抓**中央氣象署(CWA)開放資料的警報/特報**(颱風警報、豪雨特報、低溫/高溫特報)。台灣真的發警報時 → 觸發全服 **颱風假樂透 / 災難** 大事件(§17.2),與現實同步、跟直播社群一起經歷。CWA opendata **免費**(政府開放資料,註冊拿授權碼)。稀有故不靠它撐日常;情緒鉤子最強。
  - **為何只取警報層**:①颱風一年才幾次,只用真實會讓系統 95% 時間睡著 ②選址 floodRisk 差異若全台一套真實天氣就沒意義 ③**氣象預報付費機制**若接真實天氣就廢掉(玩家直接 Google CWA)→ 故真實只做「大事件觸發」,日常留給模擬層。
- **模擬層(微觀 / 每日 / 各區獨立隨機)** — 日常各區天氣**模擬**,world tick 用 **deterministic seed** 演進(可控 + 防狂點重抽 + 可測試)。**天氣上限只到大雨/梅雨、不含颱風級**(颱風保留真實層 §17.1a → 避免各區自己擲出颱風的穿幫)。維持客流變化、選址 floodRisk 差異、付費**氣象預報**(預報模擬層 → 玩家 Google 不到 → 技巧軸成立)。
- **兩條獨立軸**(設計關鍵,真實層 severity 由 CWA 警報級別映射、模擬層由 seed 生):
  - `stormSeverity` — 實際風雨強度 0-10(晴 → 颱風)
  - `holidayDeclared` — 行政放假(停班停課),與 severity **高度相關但不完全一致** → 這個**落差=玩點**。真實層放假直接讀 CWA/政府公告映射;模擬層放假由 severity + 機率生。
- 天氣型別:晴 / 多雲 / 小雨 / 大雨 / 梅雨 / 颱風 / 寒流 / 酷暑(可綁季節,見 17.6)。
- **🔴 工程要點**:真實層走小排程 Lambda 抓 CWA → 寫 world 狀態,**必做快取 + fallback**(API 當機/改格式時降級為純模擬層,不可讓核心迴圈掛掉)。

### 17.1a 真實層只認「全國性重大天氣」(**已定案 2026-07-16:無需錨點縣市**)
虛構里 ↔ 真實氣象的對齊,用「只取全國級事件」一刀解決,**不需要挑錨點縣市**:

- **真實層觸發條件 = 只認全國性重大天氣** — 即**颱風警報(海上/陸上)**與**大範圍豪雨/超大豪雨特報(涵蓋多縣市)**這種本就「全台一起」的鋒面級事件。CWA 一發 → **全服天王里一起觸發**(§17.2 樂透/災難)。因為這類事件本質是全國性的,**整個台灣 = 整個天王里**,不需要把天王里對應到某個真實縣市、也不需 `world.realAnchor`。**不取單一縣市的局部警報**(那種留給模擬層各區隨機表現)。
- **日常天氣 = 各區遊戲內隨機** — 晴~大雨這種日常級由模擬層 deterministic seed 各區獨立生(§17.1),維持選址差異 + 付費氣象預報技巧軸(預報的是模擬層 → 玩家 Google 不到)。**模擬層天氣上限不含颱風級**(颱風保留真實層,避免各區自己擲出颱風的穿幫)。
- **影響 = 各區地勢** — 不論真實層颱風或模擬層大雨,`terrain` 都決定後果:溪畔區先淹、海線區先掉招牌/停電(§2.1.1)。**真實層與模擬層共用同一套 terrain 修正**、走同一條天氣事件結算路徑,不重寫。

### 17.2 招牌機制:颱風假樂透(使用者點名的核心)
四象限(放假 × 天象):
| | 天象溫和 | 天象猛烈 |
|---|---|---|
| **有放假** | 🎰 **爆滿樂透** — 全里的人在家沒事 + 不能出遠門 + 店就在巷口 → 本地客群(高齡/媽媽/游擊中年/雀友)大爆滿 | 💀 **災難** — 近乎零客人 + 淹水/停電/招牌掉 + 維修支出 |
| **沒放假** | 平常日 | 該放沒放 — 客人照上班但心浮氣躁(mood↓)、少量硬要出門 |
- 🎰 爆滿樂透的張力:**天上掉下來的旺場,但你要排得出人手才接得住** → 沒排班 = serviceLevel 崩 = 爆滿反而變負評(§staff 命門被天氣放大)。

### 17.3 天氣 → 客流(軟性,per 天氣 × per 客群)
- 每天氣型別對每客群一個乘數,進 clientflow 當新層 `weatherMod`。要點:
  - **麻將是室內避雨/避暑娛樂** → 小雨/酷暑對「宅在附近」的本地客群反而**微升**(沒別的事做 + 來吹冷氣),但酷暑 = 冷氣電費↑。
  - 大雨/淹水 → 步行/機車客群↓↓、開車客群(大戶/媽媽)較不受影響(接 location 交通係數)。
  - 寒流 → 高齡↓(怕冷不出門)、餐飲(火鍋/宵夜)↑。
  - 颱風假 + 溫和 → 本地客群爆滿、觀光客/學生(停課)pattern 位移。
- `weatherMod` × location(交通/淹水風險)交互:低窪一樓淹水時交通 access 直接切斷。

### 17.4 天氣 → 極端事件(硬性,接 §7 事件表;**已定案 2026-07-16:個別設計、持續約 2-3 天**)
每個天氣事件**個別設計**(不套統一 tick),生命週期 seed 進場後**持續約 2-3 天(真實 1:1 Asia/Taipei)**再自然結束或被玩家處置,期間在面板有專屬呈現(狀態橫幅 + 可行動按鈕),不是一擊即逝:
- **店面淹水** — 低窪/一樓/`floodRisk` 高地段觸發 → 淹水期間(~2-3 天)停業或降客 + 需派人清理抽水 + 設備泡水損壞(電動桌/冰箱);抽水機在→縮短天數/減損。
- **招牌掉落** — 強風/`windExposure` 高 → 看板損壞 → buzz/曝光↓ 直到修復 + 維修 + (娛樂化)砸到路人賠償。
- **停電** — 颱風/雷雨 → 電動桌/冷氣/監視器停擺 → 客人跑;有發電機可擋(把停電天數影響歸零)。
- **天花板漏水** — 老舊店面/裝潢差 → 環境/清潔↓,持續數天要修。
- 每事件一組欄位:`type / startedAt / durationDays(seed 2-3) / severity / mitigations[已裝設備/已派防災]`。全部 deterministic seed(tick bucket)防狂點重抽。

### 17.5 玩家能動性(天氣不能是純 RNG 懲罰 → 必備對策軸;**已定案 2026-07-16**)
- **選址** — 低窪一樓(租金低但 `floodRisk` 高)vs 高樓層/高地(貴但安全)→ 加深 §選址(location `terrain`)。
- **防災設備**(equipment 型錄新增) — 抽水機/排水、發電機、防風鐵捲門、防水招牌、遮雨棚 → 各降特定天氣事件的機率/損失/持續天數。
- **🧹 員工防災準備**(取代天災險,**使用者定調:不做保險、改叫員工準備**) — **不是花錢買保險躺平,而是要排得出人手**。在 staff §任務系統(`CODEX_SPEC_staff.md`)新增**「防災準備」任務**(收招牌/堆沙包/清水溝/固定設備),颱風/豪雨來前派員工執行 → 降 event 機率與損失。**吃員工注意力**(§staff 並發懲罰)→ 天氣壞的那幾天,防災準備 vs 尖峰接客搶同一批人手 = 天氣把「排班命門」再放大一層。事前有查新聞預判 → 才排得出防災班。
- **📰 氣象預報 = 看新聞**(取代付費顧問,**使用者定調:出現在新聞、要特別去查看**) — 不付牙齒、不是設備;預報**出現在遊戲內「新聞」面板**,玩家要**主動去翻**才看得到未來 2-3 天的模擬層天氣趨勢(帶不確定性:越近越準、3 天外只給趨勢)+ 真實層颱風警報(跟現實同步)。技巧軸 = **勤不勤勞查新聞 + 據此預先派防災/囤貨/排班**(不是資訊付費差,是主動性 + 調度)。因預報的是模擬層 → 玩家 Google 不到,機制仍成立。

### 17.6 季節性(綁賽季/流年)
- 天氣機率綁月份:梅雨(5-6)、夏颱(7-9)、寒流(冬)、酷暑(7-8)→ 和風水流年一起換季 → 選址要算「這區這季容易淹嗎」。
