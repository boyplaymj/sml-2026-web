# Token 用量治理 SOP

開發新功能／服務時，如何**事前預測**、**事後實測**、**留下紀錄**的一套流程與參考基準。

> ⚠️ 本文所有 `$` 都是 **API 牌價換算值**，只用來抓比例、估相對規模、揪出暴衝任務。
> 實際付費是 Claude Max 訂閱吃到飽（月費固定），不是這些數字加總。真正的硬限制是
> **5 小時滾動視窗**與**每週配額**（帳號 A 每週三 03:00 重置）。目標是別在沒人盯的
> 時候把視窗配額燒光。

## 四步流程

### 1. 事前預測（提案時就附）
提任何做法時，先用下面【參考量表】估算，附在提案裡再確認才動工。公式：

```
預估 = Σ（各操作類型的單位成本 × 次數） + 迴圈/重試倍數
```

**一定要標紅的訊號**（過去燒爆額度都是這幾種）：
- 迴圈/批量：`for 每篇文章 / 每張圖 / 每個使用者` → 次數 × 單位成本，很快破千。
- 長脈絡：對話越長，每則要重載的 `cache_read` 越大（實測中位 **12.5 萬 tok/則**）。
- 背景/排程/webhook：沒人盯，最危險 → 一定要配「每日上限」與「模型分層」。

### 2. 開發
- **預設用最低夠用的模型**。同一則回覆 Opus vs Haiku 實測差 **15×**（$0.315 → $0.021）。
  抓取／摘要／結構化抽取 → Haiku 或 Sonnet 就夠，Opus 只留真正要推理的。
- 自動任務接**每日上限守門**＋**逐次記帳**（見 `stock-intel/budget.py` 範式）。

### 3. 事後實測
- **跨頻道/功能佔比**：`python3 tools/token-usage/report.py`
  （讀 `~/.claude/projects/**/*.jsonl` 逐則 usage，按單價加權，對到頻道名）。
- **自動化服務逐次帳本**：接 `--output-format json | budget.py --log <kind> <model>`，
  隨時 `python3 budget.py` 看當日實際 token/$（範式在 `/opt/sml/stock-intel/`）。

### 4. 記錄
- 功能完工時，把「本次開發實際用量」附一句在交接/commit 或對話。
- 治理相關的非顯而易見決策 → 寫進記憶（`feedback_token_preflight`、`reference_token_usage_report`）。

## 參考量表（校準自本機 52,661 則實際回覆，2026-07-07）

| 單位 | 典型量 | 牌價成本（比例用） | 備註 |
|---|---|---|---|
| **一則 AI 回覆**（bridge 回一次）| output 中位 683 tok、重載脈絡中位 125k | 中位 **$0.23**、平均 $0.52、90 分位 $1.15 | 成本主要被脈絡長度（cache_read）拉動 |
| 同一則：**Opus vs Haiku** | — | **$0.315 vs $0.021（15×）** | 降模型是最大槓桿 |
| **session 啟動固定開銷** | cache_creation 中位 4,584 tok | 一次性、~$0.01–0.09 依模型 | 每個新 `claude -p` / 新對話都有這地板 |
| **一篇股市情報卡**（定錨管線）| 全文~3k + 最多12圖 + prompt | Haiku ~$0.02–0.05／篇；**Opus ~$0.3–0.7**（修正前） | 已改 Haiku，見 `run_pipeline.sh` |
| **一次生圖**（Bedrock SD3.5）| — | **不吃 Claude token**（另計 Bedrock 費）；只有觸發它的那則回覆算 | 別把圖本身算進 token 預算 |
| **一個小功能開發**（一次對話）| 數十~數百則回覆 | 約 **$50–150** | |
| **一個中大型功能**（跨天多輪）| 數百~數千則 | 約 **$300–1,500+** | report.py 實測：計分後台累計佔全部 33% |

**估算範例**：「每 2 小時抓 N 篇文章各產一張情報卡」
= N 篇 × Haiku $0.03 × 每日 12 輪 ≈ N × $4.3/日。若誤用 Opus → ×15 ≈ N × $65/日（這就是之前燒爆的量級）。

## 硬事實速查
- 資料源：每則對話的 usage 一定寫進 `~/.claude/projects/<proj>/<session>.jsonl`。
- 頻道↔session 對映：`~/.claude/bridge-sessions.json`（只存各頻道**當前**session；開新對話後舊的查不到）。
- 換頻道名要帶 `User-Agent: DiscordBot (...)`，否則 Cloudflare 1010 擋。
- `entrypoint`：`sdk-cli`=bridge 自動、`cli`=本機互動。
- 單一頻道專屬 workdir 的舊 session 可回貼功能（見 report.py 的 `REPO_EXCLUSIVE`）；共用 `/opt/sml/repo` 的歷史 session 無法逐一回貼。

## 已知限制（別過度承諾）
- **事前預測永遠是估計**：LLM token 非決定性，本表只把誤差收斂到量級對、非精算。
- **事後歸屬有盲區**：頻道 ID 沒寫進 transcript，開新對話前的舊 session 只能按 repo 粗分。
  要精準到功能級，得像 stock-intel 那樣在服務端逐次記帳。
