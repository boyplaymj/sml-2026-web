# 甜甜報稅系統（Tax System）— 設計冊 v0.1

> 狀態：**決策已定案，級距已用真實資料校準（÷10），待交 Codex**。
> 校準：見 `tools/tax-system/calibrate.js`（P0 唯讀掃 `point-log` 93 活躍玩家、年化），級距採**台灣表 ÷10**（§5）。
> 一句話：把整年牙齒收入按**台灣綜合所得稅**的骨架結算，每年 5 月申報、5/31 截止，逃漏稅者逐級鎖權（接 `!jail` 三態閘門）。
> 制度用意（使用者定調）：**逼玩家去解免稅任務、盡量參與遊戲/活動；用「扶養」增加用戶互動、成為衝 VVIP 的動力。** 稅是壓力來源，扣抵是誘因出口。
> 關聯：[[每日任務 tools/daily-quest/DESIGN.md]]（免稅依據之一，其他 session 施工中）、監獄系統 `model/Jail.js` + `const/crimes.js`（審判閘門）、`sml-teeth-economy` Lambda（分類引擎沿用）。

## 1. 稅制精神與台灣稅法對照

真實台灣綜所稅的算法是一條輸送帶：**所得總額 → 減免稅額/扣除額 → 課稅所得 → 套累進稅率 → 減抵減稅額 → 應納稅**。本系統原封搬這條輸送帶，只把名詞換皮：

| 台灣綜所稅概念 | 本系統對應 | 遊戲誘因 |
|---|---|---|
| 各類所得（10 類） | 牙齒收入按 `reason` 分類（§3，後台逐項可調） | — |
| 免稅額（本人 97,000） | 基本免稅額 9,700（人人有） | 保護輕度玩家 |
| 免稅額（扶養親屬每人 97,000） | **扶養扣除**（每名 +9,700，上限 5，需 VVIP） | 互動 + 衝 VVIP |
| 標準扣除額（131,000） | 標準扣除額 13,100 | 保護輕度玩家 |
| 薪資所得特別扣除（218,000） | **公職特別扣除** 21,800（曾任里長助理） | 參與選舉/公職 |
| 特別扣除額（其他） | **認真里民特別扣除**（簽到＋每日任務滿 6 個月） | **逼玩家解免稅任務** |
| 投資抵減 / 重購退稅（抵稅額） | **勳功偉業抵減**（排行榜最高成就，抵稅額 10%） | 拼遊戲成就 |
| 五級累進稅率（5/12/20/30/40%）＋速算扣除 | 同（§5，牙齒版，待校準） | 懲罰肝帝/富人 |
| 分離課稅（如中獎、部分所得） | **贈與分離課稅**（低固定率，§3） | 贈與仍課但輕 |
| 綜合所得稅結算申報（5 月） | `!報稅` 5 月申報、5/31 截止（§8） | — |
| 逃漏稅裁罰 | 逃稅罪，逐級鎖權（§9，接 jail） | 逼你繳 |

## 2. 稅年與時程（使用者定案）

- **稅年**：曆年制，**1/1 – 12/31**（以 `point-log` 的 `createdAt` 切）。
- **申報期**：隔年 **5/1 開放 `!報稅`**，**5/31 24:00 截止**。
  - 例：2026 年賺的牙齒 → **2027 年 5 月**申報。
- **首屆**：上線後第一個 5 月申報「前一整年」。若上線不足一年，首屆按實際起算日到 12/31 的區間（後台可設 `firstTaxYearStart`）。
- **關鍵時點**（皆走 systemd timer / 排程 Lambda，台灣時間）：
  - `5/1 00:00` — 年度結算批次：掃 `point-log` 建每人 `tax-filing`，開放申報，公告。
  - `5/25`、`5/30` — 甜甜私訊/頻道提醒未申報者（沿用 vip-reminder timer 模式）。
  - `6/1 00:00` — 截止後送審批次：未繳清者進入欠稅 → 逐級鎖權（§9）。

## 3. 所得分類 taxonomy（逐項課稅，後台可調 — 核心設計）

**使用者要求：贈與也要課稅（相對很低），且要「逐項檢視」，後台管理。**
→ 不做寫死的「應稅/免稅」二分，而是一張**後台可編輯的分類表** `sweetbot-tax-class`：把每個 `reason` 對應到一個**所得類別**，每類別掛一種**稅務處理**。分類引擎沿用線上 Lambda `sml-teeth-economy` 的 `CATEGORY_RULES`（reason 關鍵字 → 類別），再疊一層稅務屬性。

### 所得類別 → 稅務處理（預設，全部後台可改）

| 類別 key | 台灣對應 | 稅務處理 | 涵蓋 reason（範例） |
|---|---|---|---|
| `contest` 競技競賽所得 | 競技競賽及機會中獎 | **併入綜合所得（累進）** | 賓果獎勵/賓果首殺/骰寶中獎/BJ贏/BJM贏/四連環贏/猜數字贏/1A2B獎勵/射龍門中獎/推筒子贏/戳戳樂獎項/過馬路落袋/試煉之門獎勵/龍舟獎勵/打敗屈原獎勵/競猜中獎/精華預測成功/PK守門/PK彩蛋/抽籤中獎/猜花時間/救屈原/**賽況應援所（牙齒股市）下單・收盤結算・結算兌現・分紅** |
| `salary` 薪資所得 | 薪資所得 | **併入綜合所得（累進）** | 法官判決獎勵、（未來）公職津貼、直播發薪型發獎 |
| `other` 其他所得 | 其他所得 | **併入綜合所得（累進）** | 擁有精華獎勵、雜項獎勵 |
| `gift` 贈與所得 | 贈與（分離） | **分離課稅・低固定率（預設 2%）** | 搶到紅包、壽星獎勵、（未來）玩家間轉帳收入端 |
| `welfare` 福利/補助 | 免稅所得 | **免稅（率 0%，後台可調非 0）** | 每日登入獎勵、每日獎勵、簽到 |
| `nontax` 非所得 | 不課稅 | **不計入（返還/校正/本金）** | 退款類（賓果申訴退款/競猜退款）、系統校正（負債加回/超過限額扣回）、指令給予（管理員補償，預設 nontax，後台可改 salary 課稅） |

> **淨額原則（使用者定案）**：每一「累進類別」取**全年淨額**＝`Σ variation`（該類別所有正負相加），只有**淨正**才計入所得。賭場的「骰寶中獎(+)／骰寶下注(−)」自動相抵 → 淨輸不課、淨贏才課，不會因流水灌爆。`gift` 類同樣取淨額。
> **綜合所得總額** ＝ `max(0, contest淨額) + max(0, salary淨額) + max(0, other淨額)`（各累進類別分別 floor 0 再加總，避免用某類的虧損沖掉另一類的所得——比照台灣各類所得不可跨類為負）。

### 後台逐項檢視（§11 管理頁）
- 一張表列出**近一年出現過的所有 `reason` + 目前歸類 + 該類稅務處理 + 年度總額**，管理員可逐筆改歸類、可新增「reason pattern → 類別」規則、可調各類稅率。
- 未命中任何規則的新 `reason` → 落入 `unclassified`（預設當 `other` 課稅但標紅，提醒管理員歸類）。**寧可先課再人工調**，避免漏網。

> **P0 校準實證**：初版 `calibrate.js` 規則漏掉「賽況應援所（牙齒股市遊戲）」，金額極大（下單 −58 萬/收盤 +38 萬/兌現 +19 萬/分紅 +6.8 萬/年化），本該 `contest` 卻掉進 `other` → **正是這張後台表存在的理由**。實作時 `tax-class` 預載規則須含股市各 reason；新遊戲上線也要記得補一條規則。

## 4. 扶養系統（子模組，需新建）

現況：`vipLevel` 已有 VVIP=2（role 現成），但**扶養關係表不存在** → 本系統一併設計。

- **資格**：扶養人須 **`vipLevel >= 2`（VVIP 以上）**。被扶養人無限制（但見防濫用）。
- **上限**：每人最多扶養 **5 名**。
- **效果**：申報時每名有效被扶養人 → 扶養人 **免稅額 +9,700🦷**（÷10 校準；比照台灣扶養親屬免稅額）。滿 5 名 = +48,500🦷 免稅額，對中高所得玩家很有感 → 衝 VVIP + 拉人互動的誘因。
- **綁定**：`!扶養 @對象` → 對象收到確認鈕（**雙方同意**才成立，防單方面掛人頭）→ 寫 `sweetbot-dependent`。`!解除扶養 @對象` 可解。
- **防濫用**：
  1. 一名被扶養人**同一稅年只能被一人扶養**（`dependentID` 唯一）。
  2. 綁定關係須在**該稅年 12/31 前成立**才計入當年（申報時檢查 `boundAt <= 稅年底`），防 5 月現抓人頭。
  3. 被扶養人須為**活躍帳號**（該稅年有簽到/發言/遊戲紀錄任一），防養殭屍分身。
  4. 扶養人一旦掉出 VVIP，既有扶養**當年仍有效**、但無法新增（比照資格以綁定時 + 申報時雙檢，寬鬆取一，後台可調嚴格度）。

## 5. 稅額計算（台灣五級累進・牙齒版・÷10 校準）

> **級距已校準**：P0 掃 `point-log`（93 活躍玩家年化）顯示牙齒所得量級約台灣 1/10~1/20，若直接套台灣數字只有 4.3% 會被課到稅（不痛）。**採台灣表 ÷10**：約 1/3 玩家進入稅網（未計扣抵前），扣抵疊上後有做免稅任務者多歸零、只剩「肝帝不做任務」被課——正是設計意圖。全部存後台 config，可調。

### 計算輸送帶（單人・單稅年）

```
綜合所得總額            = max(0,contest) + max(0,salary) + max(0,other)
  − 免稅額              = 9,700 × (1 本人 + N 有效扶養)
  − 標準扣除額          = 13,100
  − 公職特別扣除        = 曾任里長助理 ? 21,800 : 0
  − 認真里民特別扣除    = 簽到+每日任務滿6個月 ? 15,000 : 0
  = 綜合所得淨額（課稅所得，floor 0）
  × 累進稅率 − 速算扣除 = 綜所應納稅額
  + 贈與分離稅          = max(0, gift淨額) × 2%
  = 應納稅額合計
  − 勳功偉業抵減        = 綜所應納稅額 × 10%（有最高成就者，見 §6）
  = 【實際應繳稅】（floor 0，四捨五入到整數牙齒）
```

### 五級累進（牙齒，台灣 5/12/20/30/40% ÷10 校準）

| 綜合所得淨額（🦷） | 稅率 | 速算扣除額（🦷） |
|---|---|---|
| 0 – 59,000 | 5% | 0 |
| 59,001 – 133,000 | 12% | 4,130 |
| 133,001 – 266,000 | 20% | 14,770 |
| 266,001 – 498,000 | 30% | 41,370 |
| 498,001 以上 | 40% | 91,170 |

- **課稅門檻**：綜合所得總額 ≤（免稅額＋標準扣除額）＝**單身無扶養 22,800🦷** → 課稅所得 0、**免繳**（仍申報確認 / 系統自動核定 0）。中位數玩家（年化 ~2,069🦷）遠低於門檻 → 休閒玩家不會被課。
- **速算**：應納稅 ＝ 課稅所得 × 稅率 − 速算扣除額。
- **贈與分離課稅**：不併入累進、單獨 2%（後台可調）。
- **實證稅負（÷10，年化）**：頂點玩家（毛額 ~52 萬🦷）年繳 ~10.8 萬🦷（有效稅率 ~20%，富人重稅）；全服年綜所稅收 ~47 萬🦷 + 贈與稅 ~4.6 萬🦷（= sink，利通膨）。
- **⚠️ 上線前複校**：目前基於 17.6 天（7 月，含端午龍舟活動可能偏高）×20.7 年化，樣本薄。**累積更多月份後重跑 `calibrate.js` 微調**；首屆申報（2027/5）前資料已厚，屆時定稿。

## 6. 勳功偉業抵減 → 成就轉接器（子模組，需新建）

現況：**各遊戲排行榜各存各的、無統一成就系統**（拳王 `sweetbot-rpt-record`、龍舟 `sweetbot-dragonboat-record`、骰寶 `sweetbot-sicbo-record`、二八杠 `sweetbot-inbetween-reward`、麻將賽季 Firestore `sml_season_stats`…）。

- **本系統做一張登錄表 `sweetbot-achievement`**：`pk=discordID`, `sk=achievementKey`，欄位 `title / source（哪個榜）/ tier（金銀銅/rank）/ earnedAt / taxYear`。
- **v1 認定**（後台可調清單）：指定幾個榜的頂點寫入 → 拳王紀錄保持人、龍舟 TOP3、骰寶/二八杠榜首、麻將賽季 rank 1、過馬路距離榜首… 每個 = 一枚勳功。
- **抵減規則**：該稅年**擁有任一勳功** → **綜所應納稅額 −10%**（封頂 10%，不疊加；後台可改成分級）。
- **未來擴充（呼應使用者第 4 點「未來所有排行榜都與報稅連動」）**：各遊戲結算時 fire-and-forget 寫 `sweetbot-achievement`（比照 `QuestTracker.track` 的埋點模式），報稅只讀這張表，不用回頭改各遊戲。**先做轉接器介面，各遊戲逐步接。**

## 7. 認真里民特別扣除 → 簽到＋每日任務（使用者定案）

- **資格**：該稅年內，**簽到 + 每日任務累計達成滿 6 個月**（後台可調門檻天數，預設 180 個「達成日」，非連續亦可，或設連續，見待定）。
- **資料源（雙依據）**：
  - 簽到：`sweetbot-daily-checkin`（PK=discordID, SK=year/month/day）— **現成**，可即刻查歷史。
  - 每日任務：`sweetbot-daily-quest`（PK=discordID, SK=YYYY-MM-DD，含 `streak`/`quests[].done`）— **其他 session 施工中**。
- **v1 過渡**：每日任務未上線前，**先只認簽到**（滿 180 日）；每日任務上線後，改為「簽到 **或** 當日解滿任務」任一即算一個達成日，門檻不變。此開關放後台（`liyminUseQuest: false→true`）。
- **效果**：綜合所得淨額 **−15,000🦷** 特別扣除（÷10 校準，後台可調）。這是「逼玩家解免稅任務」的主誘因——認真解任務一整年，報稅直接少一大截（15,000 ≈ 把一個年化 ~5 萬所得的中段玩家壓回免稅門檻附近）。

## 8. 申報流程 UI（`!報稅` 一指令開面板，之後全按鈕）

沿用「一指令開場、之後全互動、同訊息重繪不洗版」慣例（範本 `InBetween.js`）。

1. **`!報稅`**（僅 5/1–5/31 開放；平時回「非申報期，你的即時稅務試算：…」唯讀）。
2. 面板 embed 顯示該稅年**稅單**：
   - 各所得類別淨額（contest / salary / other / gift）
   - 減項明細（免稅額×人數、標準扣除、公職、認真里民）
   - 課稅所得 → 套級距 → 綜所稅 + 贈與稅 − 勳功抵減 = **應繳稅額**
   - 目前牙齒餘額、繳稅後餘額。
3. **按鈕**：
   - `📋 逐項檢視`：展開該稅年收入按 reason 明細（透明、可服眾）。
   - `👨‍👩‍👧 扶養申報`：列出已綁定被扶養人、可即時 `!扶養` 綁定（VVIP 才有此鈕）。
   - `✅ 確認申報並繳納`：從餘額扣稅（reason=`繳稅`，計入 sink）→ 狀態 `paid`、發「完稅」提示。
   - `🧾 分期 / 欠稅`：餘額不足時，可部分繳、掛欠稅（見 §9 後果）。
4. **冪等**：`sweetbot-tax-filing` 的 `status` 防重繳；已 `paid` 再開面板顯示收據。
5. **自動核定**：截止時仍未申報者，系統按已算稅單**視同核定**（不因不申報就免稅），逾期未繳即欠稅。

## 9. 審判 / 逃漏稅（逐級鎖權，接 `!jail`）

**使用者定案：要有繳納期限、逾期「逐一鎖權限」。** → 兩級階梯，接現成三態閘門 `CommonUtil.passCriminalGate()`（allow / block / only 已實作，遊戲入口零改動）。

### 新增罪名（`const/crimes.js` 加 id:5）

| 階段 | 觸發 | 身分組 | 鎖什麼（criminalAccess） | 解除 |
|---|---|---|---|---|
| **欠稅**（輕） | 6/1 未繳清 | 新 role `欠稅戶`（需新建） | `block` **高收益博弈類**（賓果/骰寶/BJ/推筒子/射龍門/競猜…切斷再撈牙齒的路，逼你繳）；社交/簽到/領每日仍 `allow` | 補繳全部欠稅即自動解除 |
| **逃稅罪**（重） | 欠稅逾 30 天仍未繳（後台可調），或惡意零申報 | 新 role `逃稅犯`（需新建，接 jail sentence） | 幾乎全 `block`，只剩 `only` 的勞動贖罪類（比照洗牙罪的重刑閘門） | 補繳 **或** 勞動贖罪（湊滿勞動種類數，沿用 `jail.recordLabor` + 假釋機制） |

- **接法**：逃稅罪走 `Jail` 現成 `sentence` 檔案（`sweetbot-viewer.sentence`）+ `limitedTimeRole.giveRole` + 假釋清檔。欠稅戶較輕，可只掛 role + 在 `passCriminalGate` 讀「欠稅」旗標，不必進完整刑案流程。
- **前科**：逃稅比照 `crimeCounts` 累犯加重（連年逃稅 → 保釋/贖罪門檻加重，沿用 `BAIL_MULTIPLIER_LADDER`）。
- **補稅即赦**：欠稅戶只要 `!報稅` 補繳到 `paid` → 即時解 role、恢復遊戲權限（比照假釋 `expiredNow`）。
- **勞動贖罪**：真的繳不出（餘額不足又不肯玩）→ 走監獄現成「社會勞動」湊假釋，象徵「易服勞役」。

### 送審批次（6/1 timer）
掃 `sweetbot-tax-filing` 該稅年 `status != paid && 應繳 > 0` → 掛「欠稅戶」role + 記 filing `status=overdue`。逾 30 天再掃一次升級「逃稅犯」。

### ⭐ 隨賺隨記帳本（P0 發現，取代「年終掃流水」）

**問題**：P0 發現 `point-log` 才從 2026-06-28 開始記帳、且未來會越長越肥；「5/1 掃一整年原始流水」既依賴長期保留、又要掃越來越大的表。

**方案**：在既有發牙函式 `ViewerDetailDAO.givePoint(ids, amount, col, reason)` 內**加一行 fire-and-forget**：`pointType==='point'` 時 → 依 `tax-class` 分類 → `ADD` 累加進 `sweetbot-tax-ledger` 的 `(discordID, 當年)` 該類別（原子累加，仿牙齒經濟 rollup / QuestTracker 埋點模式）。
- **好處**：申報只讀一列；與 point-log 保留策略**完全解耦**；即時、無批次掃描。
- **相容**：`givePoint` 是幾乎所有發牙的單一出口 → 埋一處覆蓋全遊戲；埋點全 catch、絕不影響發牙主流程。
- **回填**：ledger 上線前已發生的收入（2026-06-28 起）用一次性腳本從 point-log 回填當年 ledger（首屆用）。
- **年度結算批次（5/1）**：改為讀 `tax-ledger` 快照 → 算稅 → 寫 `tax-filing`（不再逐人掃 point-log）。

> **保留依賴（次要備援）**：即使改帳本制，仍建議 `point-log` **保留 ≥13 個月**（目前 TTL 已 DISABLED ✓）供逐項稽核/爭議查證；勿在未評估下對 point-log 開短 TTL。

## 12. 💰 成本控管（遵循 tools/COST_CONTROL.md）

- 若日後加 LLM（如 AI 生成稅務諮詢對話），回本規範補「四件套」。
